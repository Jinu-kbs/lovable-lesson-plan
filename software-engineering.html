1소프트웨어 엔지니어링이란?

소프트웨어 엔지니어링은 소프트웨어를 체계적·효율적·신뢰성 있게 개발하는 공학적 접근입니다. 단순히 동작하는 코드를 작성하는 것을 넘어, 오래 유지할 수 있고 다른 사람과 함께 발전시킬 수 있는 코드를 만드는 방법론의 집합입니다.

탄생 배경: 1968년 나토(NATO) 컨퍼런스에서 "소프트웨어 위기(Software Crisis)"를 논의하면서 SW 엔지니어링이 독립 학문으로 자리 잡았습니다. 당시 많은 프로젝트가 예산 초과, 기한 지연, 품질 미달로 실패했고, 이를 해결하기 위한 체계적 접근이 필요했습니다.

SW 엔지니어링의 4가지 핵심 목표

🔧

📈

💰

소프트웨어 엔지니어링은 코딩 스타일 규칙에서 시작해, 시스템 설계 원칙, 팀 협업 방법론, 품질 보증 프로세스까지 넓은 범위를 다룹니다.

2왜 배워야 하나?

AI가 코드를 대신 써준다면 SW 엔지니어링을 알아야 할까요? 답은 더욱 그렇다입니다.

"AI는 당신이 원하는 것을 만들어 주지만, 올바른 방식으로 만들지는 않습니다. 올바른 방향을 잡는 것은 여전히 당신의 몫입니다."

바이브코딩의 함정

⚠️ AI가 만든 나쁜 코드

  • 동작은 하지만 이해하기 어렵다
  • 중복 로직이 여기저기 흩어진다
  • 한 군데 고치면 다른 곳이 깨진다
  • 테스트가 없어 변경이 두렵다
  • 3개월 후 아무도 이 코드를 못 읽는다

✅ SE 원칙이 적용된 코드

  • 코드만 읽어도 의도가 파악된다
  • 각 함수·클래스가 명확한 역할을 갖는다
  • 테스트가 변경 안전망이 되어준다
  • 새 기능을 기존 코드 수정 없이 추가한다
  • 팀원이 빠르게 온보딩할 수 있다

AI와 SE 원칙의 시너지

SW 엔지니어링 원칙을 이해하면 AI에게 더 좋은 코드를 지시할 수 있습니다. AI는 내가 요청하는 대로 만들어 줍니다. 내가 "SOLID 원칙을 지켜서 리팩토링해줘"라고 요청할 수 있는 사람과 그냥 "코드 써줘"라고 요청하는 사람의 결과물은 완전히 다릅니다.

핵심 인사이트

AI 도구는 SW 엔지니어링 지식을 가진 개발자에게 슈퍼파워를 줍니다. 원칙을 모르는 사람에게는 빠르게 기술 부채를 쌓는 도구가 됩니다.

3핵심 설계 원칙 개요

수십 년간 소프트웨어 개발 경험에서 증류된 핵심 원칙들입니다. 이 원칙들은 특정 언어나 프레임워크에 종속되지 않으며, AI가 만드는 코드를 평가할 때도 동일하게 적용됩니다.

S

SOLID

객체지향 설계의 5가지 황금 원칙. 유지보수성과 확장성의 기반.

D

DRY

Don't Repeat Yourself. 중복을 제거하고 단일 출처를 유지하라.

K

KISS

Keep It Simple, Stupid. 단순한 해결책이 최선이다.

Y

YAGNI

You Aren't Gonna Need It. 지금 필요하지 않은 기능은 만들지 마라.

4SOLID 원칙

로버트 C. 마틴(Uncle Bob)이 정리한 객체지향 설계의 5가지 원칙입니다. 함수형 프로그래밍이나 AI 생성 코드에도 동일하게 적용할 수 있습니다.

S

Single Responsibility Principle — 단일 책임 원칙

클래스(또는 함수)는 하나의 책임만 가져야 합니다. 변경 이유가 하나여야 한다는 의미이기도 합니다.

나쁜 예: UserManager가 사용자 저장 + 이메일 발송 + 로그 기록을 모두 담당
좋은 예: UserRepository(저장), EmailService(이메일), AuditLogger(로그) 분리

O

Open/Closed Principle — 개방/폐쇄 원칙

소프트웨어는 확장에는 열려 있고, 수정에는 닫혀 있어야 합니다. 새 기능을 추가할 때 기존 코드를 건드리지 않는 것이 목표입니다.

나쁜 예: 새 결제 방법마다 if (paymentType === 'kakao') 분기 추가
좋은 예: PaymentStrategy 인터페이스를 구현하는 새 클래스만 추가

L

Liskov Substitution Principle — 리스코프 치환 원칙

자식 클래스는 부모 클래스를 완전히 대체할 수 있어야 합니다. 부모 타입을 사용하는 코드에서 자식 타입을 써도 동작이 올바르게 유지되어야 합니다.

나쁜 예: Square가 Rectangle을 상속하지만 setWidth/setHeight 동작이 다름
좋은 예: 부모의 계약(Contract)을 자식이 완전히 이행

I

Interface Segregation Principle — 인터페이스 분리 원칙

클라이언트가 사용하지 않는 인터페이스에 의존하도록 강요하지 마세요. 큰 인터페이스 하나보다 작은 인터페이스 여러 개가 낫습니다.

나쁜 예: Animal 인터페이스에 fly(), swim(), run()이 모두 있어 새가 swim()을 구현해야 함
좋은 예: Flyable, Swimmable, Runnable로 분리

D

Dependency Inversion Principle — 의존성 역전 원칙

고수준 모듈이 저수준 모듈에 직접 의존하지 말고, 둘 다 추상화(인터페이스)에 의존해야 합니다.

나쁜 예: OrderService가 MySQLDatabase를 직접 참조
좋은 예: OrderService가 DatabaseInterface에 의존하고, MySQLDatabase가 이를 구현

💡 바이브코딩에서 SOLID 적용하기

AI에게 코드를 요청할 때 "SOLID 원칙을 준수하여 작성해줘"라고 명시하면 AI가 더 구조적인 코드를 생성합니다. 특히 Single Responsibility와 Dependency Inversion은 테스트 작성 및 리팩토링 시 큰 도움이 됩니다.

5DRY · KISS · YAGNI

DRY Don't Repeat Yourself — 반복하지 마라

"모든 지식은 시스템 내에서 단 하나의 명확한 표현을 가져야 한다." — The Pragmatic Programmer

같은 로직이 여러 곳에 있으면 수정 시 모두 찾아서 바꿔야 합니다. 하나를 빠뜨리면 버그가 생깁니다.

❌ 나쁜 예 — 중복 코드
// 두 함수가 동일한 로직을 갖는다
function validateUserEmail(email) {
  return email.includes('@') && email.includes('.');
}

function checkAdminEmail(email) {
  return email.includes('@') && email.includes('.'); // 중복!
}

// 이메일 검증 규칙이 바뀌면 두 곳을 다 고쳐야 한다
✅ 좋은 예 — DRY 적용
// 단일 출처 (Single Source of Truth)
function isValidEmail(email) {
  return email.includes('@') && email.includes('.');
}

// 재사용
function validateUserEmail(email) { return isValidEmail(email); }
function checkAdminEmail(email) { return isValidEmail(email); }
KISS Keep It Simple, Stupid — 단순하게 유지하라

대부분의 시스템은 단순하게 유지될 때 가장 잘 작동합니다. 복잡성은 버그의 온상입니다. 영리한 코드보다 읽기 쉬운 코드가 낫습니다.

❌ 나쁜 예 — 불필요하게 복잡
const getAge = (birthYear) =>
  new Date().getFullYear() - birthYear > 18 ? true : false;
// true/false를 직접 반환하는 삼항 연산자는 불필요
✅ 좋은 예 — KISS 적용
const isAdult = (birthYear) =>
  new Date().getFullYear() - birthYear >= 18;
// 비교 연산 자체가 이미 boolean을 반환한다
YAGNI You Aren't Gonna Need It — 필요해질 때 만들어라

"지금 필요하지 않은 기능은 만들지 마라. 나중에 필요하면 그때 만들어라."

미래를 대비해 미리 만든 기능의 80%는 실제로 사용되지 않습니다. 그 코드는 유지보수 부담만 됩니다. AI와 함께 작업할 때 특히 주의하세요. AI는 요청하지 않아도 "유용할 것 같은" 기능을 추가하는 경향이 있습니다.

⚠️ AI와 YAGNI

AI 도구는 요청보다 많은 코드를 생성하는 경향이 있습니다. "딱 필요한 것만" 이라고 명시하거나, 생성된 코드에서 당장 사용하지 않는 부분을 제거하는 습관을 들이세요.

6개발 방법론

폭포수 vs 애자일

🌊 폭포수(Waterfall)

  • 요구사항 → 설계 → 개발 → 테스트 → 배포 순차 진행
  • 처음부터 전체를 설계
  • 단계 간 피드백이 늦다
  • 변경이 어렵고 비용이 크다
  • 대형 정부 프로젝트, 임베디드에 적합

🔄 애자일(Agile)

  • 짧은 스프린트(1~4주) 반복
  • 지속적인 고객 피드백 반영
  • 변경 수용을 가치로 삼는다
  • 작동하는 소프트웨어를 자주 배포
  • 스타트업, 웹/앱 서비스에 적합

스크럼(Scrum) 기초

애자일의 대표적 프레임워크입니다. 바이브코딩 팀 프로젝트에도 바로 적용할 수 있습니다.

계획
스프린트 계획 (Sprint Planning)

이번 스프린트에서 무엇을 만들지 목표와 태스크를 정한다. 보통 1~2시간.

일일
데일리 스탠드업 (Daily Standup)

어제 한 것, 오늘 할 것, 막힌 것 15분 이내 공유. 서서 진행해 짧게 유지한다.

리뷰
스프린트 리뷰 (Sprint Review)

스프린트 결과물을 이해관계자에게 데모. 피드백을 수집한다.

회고
스프린트 회고 (Retrospective)

팀 프로세스를 개선한다. 잘된 것, 개선할 것, 시도할 것을 논의.

TDD — 테스트 주도 개발

테스트를 먼저 작성하고, 그 테스트를 통과하는 코드를 나중에 작성하는 개발 방식입니다.

7코드 품질

Robert C. Martin의 Clean Code에서 정리한 핵심 원칙들입니다. AI 생성 코드를 리뷰할 때도 동일 기준으로 적용하세요.

1. 이름 짓기 — 의도가 드러나야 한다

❌ 나쁜 예
let d; // 며칠이 지났는지? 날짜? 데이터?
function p(u, s) { /* ... */ } // p가 뭔지 알 수 없다
✅ 좋은 예
let daysSinceCreation;
function countActiveUsersByStatus(users, status) { /* ... */ }

2. 함수 — 하나의 일만, 20줄 이하

❌ 나쁜 코드
function p(u, s) {
  let r = 0;
  for(let i = 0; i < u.length; i++) if(u[i].a === s) r++;
  return r;
  // 변수명이 모두 단 1글자, 로직 파악 불가
}
✅ 클린 코드
/**
 * 특정 상태의 활성 사용자 수를 반환합니다.
 * @param {User[]} users - 사용자 배열
 * @param {string} status - 조회할 계정 상태
 * @returns {number} 해당 상태의 사용자 수
 */
function countActiveUsersByStatus(users, status) {
  return users.filter(user => user.accountStatus === status).length;
}

3. 주석 — 코드가 설명하지 못하는 것만

  • 필요한 주석: 비즈니스 규칙의 이유, 외부 시스템과의 계약, 성능상 결정 근거
  • 불필요한 주석: 코드를 그대로 설명하는 주석 (// i를 1 증가시킨다)
  • 최선: 코드 자체가 문서가 되도록 이름을 잘 짓는 것

4. 에러 처리 — 방어적으로

✅ 에러 처리 패턴
async function getUserProfile(userId) {
  if (!userId) throw new Error('userId는 필수값입니다');

  try {
    const user = await userRepository.findById(userId);
    if (!user) return null; // null을 반환하되 명시적으로

    return {
      id: user.id,
      name: user.name,
      email: user.email
    };
  } catch (error) {
    console.error(`[getUserProfile] userId=${userId}`, error);
    throw new Error('사용자 정보 조회에 실패했습니다');
  }
}

📋 코드 품질 체크 습관

AI가 코드를 생성한 후 "이 코드에서 클린 코드 원칙을 위반하는 부분을 찾아줘"라고 요청해보세요. AI 스스로 자신이 만든 코드의 문제를 찾고 개선안을 제시해 줄 수 있습니다.

8아키텍처 패턴

MVC — Model-View-Controller

웹 애플리케이션에서 가장 널리 쓰이는 패턴입니다. 관심사를 3가지로 분리합니다.

Model

데이터 &
비즈니스 로직

↔
알림/업데이트

Controller

입력 처리 &
중간 조율

↔
렌더 지시

View

화면 표시 &
사용자 입력

  • Model: 데이터 구조와 비즈니스 로직. DB와 통신하며 규칙을 강제한다.
  • View: 사용자에게 보이는 UI. Model의 데이터를 화면으로 렌더링한다.
  • Controller: 사용자 입력을 받아 Model을 업데이트하고 View에 반영한다.

클린 아키텍처 — 레이어드 구조

Uncle Bob이 제안한 패턴으로, 비즈니스 로직을 외부 의존성(DB, UI, 프레임워크)으로부터 분리합니다.

최외곽
Presentation Layer (UI)

React, Vue 등 UI 프레임워크. 화면 렌더링과 사용자 입력 처리. 자주 바뀌는 레이어.

응용
Application Layer (Use Cases)

애플리케이션 특화 비즈니스 규칙. "사용자 회원가입", "주문 처리" 같은 유즈케이스.

도메인
Domain Layer (Business Rules)

핵심 비즈니스 엔터티와 규칙. 프레임워크와 완전히 독립적. 가장 안정적인 레이어.

인프라
Infrastructure Layer (DB, API)

9테스팅 전략

테스트 피라미드

테스트를 어느 레벨에서 얼마나 작성할지 가이드하는 모델입니다.

E2E 테스트
느리고 비싸지만 전체 흐름 검증
통합 테스트
모듈 간 연결, API, DB 연동
단위 테스트
가장 많이 작성. 빠르고 저렴.

단위 테스트 (Unit Test)

가장 작은 단위(함수, 메서드)를 독립적으로 테스트합니다. 빠르게 실행되고 작성 비용이 낮습니다.

Jest 단위 테스트 예시
// isAdult.test.js
import { isAdult } from './utils';

describe('isAdult', () => {
  test('만 18세는 성인이다', () => {
    const currentYear = new Date().getFullYear();
    expect(isAdult(currentYear - 18)).toBe(true);
  });

  test('만 17세는 성인이 아니다', () => {
    const currentYear = new Date().getFullYear();
    expect(isAdult(currentYear - 17)).toBe(false);
  });

  test('출생연도가 없으면 false를 반환한다', () => {
    expect(isAdult(null)).toBe(false);
  });
});

통합 테스트 (Integration Test)

여러 모듈이 함께 작동하는지 테스트합니다. API 엔드포인트, DB 연동 등을 검증합니다.

API 통합 테스트 예시
// users.integration.test.js
import request from 'supertest';
import app from '../app';

describe('POST /api/users', () => {
  test('유효한 데이터로 사용자를 생성한다', async () => {
    const response = await request(app)
      .post('/api/users')
      .send({ name: '김테스트', email: 'test@example.com' });

    expect(response.status).toBe(201);
    expect(response.body.email).toBe('test@example.com');
  });
});

E2E 테스트 (End-to-End Test)

실제 브라우저에서 전체 사용자 흐름을 테스트합니다. Playwright, Cypress 등을 사용합니다.

Playwright E2E 테스트 예시
// signup.e2e.test.js (Playwright)
import { test, expect } from '@playwright/test';

test('회원가입 전체 플로우', async ({ page }) => {
  await page.goto('https://myapp.com/signup');
  await page.fill('[name="email"]', 'user@example.com');
  await page.fill('[name="password"]', 'SecurePass123!');
  await page.click('[type="submit"]');

  await expect(page).toHaveURL('/dashboard');
  await expect(page.locator('h1')).toContainText('환영합니다');
});

AI 보조 테스팅

Claude Code에게 테스트 작성 요청하는 법

  • "이 함수에 대한 Jest 단위 테스트를 엣지 케이스 포함해서 작성해줘"
  • "경계값 분석(boundary value analysis)을 적용한 테스트 케이스를 만들어줘"
  • "이 API 엔드포인트에 대한 통합 테스트를 supertest로 작성해줘"
  • "현재 테스트 커버리지를 분석하고 누락된 케이스를 찾아줘"

10CI/CD 기초

CI (Continuous Integration)는 코드를 자주 병합하고 자동으로 테스트하는 것이고, CD (Continuous Delivery/Deployment)는 검증된 코드를 자동으로 프로덕션에 배포하는 것입니다.

CI/CD 파이프라인

→

→

GitHub Actions 예시

GitHub 저장소에 .github/workflows/ci.yml을 추가하면 Push마다 자동으로 실행됩니다.

기본 CI/CD 워크플로우
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    name: 테스트 실행
    runs-on: ubuntu-latest
    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v3

      - name: Node.js 설정
        uses: actions/setup-node@v3
        with:
          node-version: '20'
          cache: 'npm'

      - name: 의존성 설치
        run: npm ci

      - name: 린트 검사
        run: npm run lint

      - name: 단위 테스트
        run: npm test -- --coverage

  deploy:
    name: 프로덕션 배포
    needs: test          # test 잡 성공 후에만 실행
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v3

      - run: npm ci

      - name: 빌드
        run: npm run build

      - name: GitHub Pages 배포
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist

CLAUDE.md에 추가할 코드 품질 섹션
## 코드 품질 원칙

### 설계 원칙
- SOLID 원칙 준수 (특히 SRP, OCP)
- DRY: 동일 로직은 반드시 추출하여 재사용
- KISS: 단순한 해결책을 선택한다. 영리한 코드보다 읽기 쉬운 코드
- YAGNI: 현재 필요한 기능만 구현. 미래 기능은 미리 만들지 않는다

### 함수/클래스 규칙
- 함수는 단일 책임, 20줄 이하 유지
- 모든 공개(public) 함수에 JSDoc 주석 필수
- 변수/함수명은 의도가 드러나도록 (약어 지양)
- 에러는 조용히 무시하지 말고 적절히 처리

### 테스트
- 새 함수 작성 시 Jest 단위 테스트 함께 작성
- 변경 시 기존 테스트가 통과되는지 확인
- 엣지 케이스 (null, 빈 배열, 경계값) 포함

### 커밋 전 체크
- ESLint 오류 없음
- 기존 테스트 모두 통과
- 콘솔 로그 제거

🏆 최종 목표

SW 엔지니어링 원칙은 AI 도구를 사용할수록 더 중요해집니다. AI가 빠르게 코드를 만들어 줄수록, 그 코드를 평가하고 방향을 잡는 여러분의 안목이 결과물의 품질을 결정합니다. 원칙을 이해한 바이브코더가 가장 강력한 개발자입니다.
