1. 프롬프트 체이닝과 멀티스텝 전략

학습 목표

  • 복잡한 작업을 단계별로 나누는 "프롬프트 체이닝" 개념을 이해한다
  • 순차, 분기, 병렬 체이닝 패턴을 구분하고 상황에 맞게 적용한다
  • 바이브코딩에서 멀티스텝 전략으로 더 정교한 결과를 만들어낸다

핵심 개념

프롬프트 체이닝이란?

하나의 거대한 프롬프트로 모든 것을 해결하려는 대신, 여러 개의 작은 프롬프트를 연결하여 복잡한 작업을 수행하는 전략입니다.

┌─────────────────────────────────────────────────────────┐
│              프롬프트 체이닝 기본 구조                      │
│                                                          │
│  [프롬프트 1]  →  [결과 1]  →  [프롬프트 2]  →  [결과 2]   │
│     (기획)         (기획서)      (디자인)        (UI 설계)  │
│                                      ↓                    │
│                               [프롬프트 3]  →  [최종 결과]  │
│                                  (코딩)        (완성 앱)   │
└─────────────────────────────────────────────────────────┘

왜 체이닝이 필요한가?

단일 프롬프트 방식체이닝 방식
"학급 관리 앱 만들어줘"1단계: 기능 목록 작성 → 2단계: UI 설계 → 3단계: 코드 생성
AI가 모든 것을 한 번에 결정각 단계를 검토하고 수정 가능
결과가 불안정하고 예측 어려움단계별 품질 관리 가능
수정이 어려움 (전체 재생성)특정 단계만 수정 가능
컨텍스트 윈도우 낭비각 단계에 집중된 컨텍스트

3가지 체이닝 패턴

패턴 1: 순차 체이닝 (Sequential)

앞 단계의 결과가 다음 단계의 입력이 되는 가장 기본적인 패턴입니다.

[A] → [B] → [C] → [D]
기획    설계    개발    테스트

적합한 상황:

  • 앱 개발 (기획 → 디자인 → 코딩 → 배포)
  • 글쓰기 (개요 → 초안 → 수정 → 최종)
  • 데이터 처리 (수집 → 정제 → 분석 → 시각화)

패턴 2: 분기 체이닝 (Branching)

하나의 결과에서 여러 갈래로 나뉘는 패턴입니다.

              ┌→ [B1] 모바일 UI
[A] 기획서 → ├→ [B2] 데스크탑 UI
              └→ [B3] API 설계
                      ↓
              [C] 통합 및 연결

적합한 상황:

  • 반응형 웹 디자인 (모바일 + 데스크탑 동시 설계)
  • 프론트엔드/백엔드 분리 개발
  • 다국어 콘텐츠 동시 생성

패턴 3: 병렬 체이닝 (Parallel)

독립적인 작업을 동시에 진행하는 패턴입니다.

[A1] 헤더 컴포넌트 ─┐
[A2] 메인 컴포넌트 ─┼→ [B] 통합
[A3] 푸터 컴포넌트 ─┘

적합한 상황:

  • UI 컴포넌트를 각각 만들고 통합
  • 여러 섹션의 콘텐츠를 동시에 작성
  • 독립적인 기능 모듈을 각각 개발

기본 프롬프트 (공통 기획)

독서 기록 관리 앱을 만들려고 합니다.
핵심 기능: 책 등록, 독서 메모, 독서량 통계

이 기획을 기반으로 두 가지 버전의 프롬프트를 만들어주세요:

버전 A — 초등학생용:
- 큰 글씨, 귀여운 아이콘
- 별점 대신 이모지 평가 (재미있어요/보통이에요/어려웠어요)

- 월간 독서 통계 차트

실전 팁

체이닝의 핵심 규칙: 각 단계의 결과를 반드시 검토한 후 다음 단계로 넘어가세요. 1단계 결과가 잘못되면 이후 모든 단계가 틀어집니다.

  • 단계 사이에 "검증 프롬프트"를 끼워넣으세요: "위 결과에서 빠진 것이나 개선할 점이 있다면 알려줘"
  • 각 단계의 출력 형식을 통일하세요: 다음 단계에서 입력으로 사용하기 쉬운 형식(표, 목록, JSON)으로 출력
  • 단계가 3개를 넘어가면 문서로 정리하세요: 각 단계의 입력/출력/검증 기준을 기록해두면 재사용 가능

주의사항

  • 과도한 분할 주의: 너무 세분화하면 각 단계 사이에서 맥락이 끊길 수 있습니다. 3~5단계가 적정합니다.
  • 컨텍스트 전달 누락: 다음 단계로 넘어갈 때 이전 단계의 결과를 반드시 포함하세요. "아까 만든 거 기반으로"라고만 하면 AI가 정확한 맥락을 못 잡을 수 있습니다.
  • 중간 결과 미검토: 1단계 결과를 확인하지 않고 바로 2단계로 넘어가면, 잘못된 기초 위에 건물을 짓는 것과 같습니다.
  • 단일 대화에서 모든 체인 실행: 컨텍스트 윈도우를 넘어설 수 있으므로, 긴 체인은 새 대화로 나누는 것을 고려하세요.

2. 컨텍스트 윈도우 최적화

학습 목표

  • 컨텍스트 윈도우의 개념과 제약을 정확히 이해한다
  • 토큰 효율을 높이는 전략을 실전에 적용한다
  • "Lost in the Middle" 현상을 이해하고 핵심 정보 배치를 최적화한다
  • 바이브코딩에서 컨텍스트 손실을 방지하는 방법을 익힌다

핵심 개념

컨텍스트 윈도우란?

AI 모델이 한 번에 "볼 수 있는" 텍스트의 최대 크기입니다. 책상 위에 펼칠 수 있는 서류의 최대 양이라고 생각하면 됩니다.

┌──────────────────────────────────────────────────────┐
│              컨텍스트 윈도우 구조                       │
│                                                       │
│  ┌───────────────────────────────────────────────┐   │
│  │  시스템 프롬프트 (역할, 규칙)     [~500 토큰]   │   │
│  ├───────────────────────────────────────────────┤   │
│  │  이전 대화 히스토리                [가변]       │   │
│  ├───────────────────────────────────────────────┤   │
│  │  현재 입력 (사용자 프롬프트)       [가변]       │   │
│  ├───────────────────────────────────────────────┤   │
│  │  출력 공간 (AI 응답)              [가변]       │   │
│  └───────────────────────────────────────────────┘   │
│                                                       │
│  ※ 모든 영역의 합 = 컨텍스트 윈도우 크기 이내         │
└──────────────────────────────────────────────────────┘

"Lost in the Middle" 현상

AI 모델은 긴 텍스트에서 처음 부분과 끝 부분에 높은 주의를 기울이고, 중간 부분의 정보를 놓치는 경향이 있습니다. 이를 "Lost in the Middle" 현상이라 합니다.

주의력 분포:

높음 ████████░░░░░░░░░░░░░░░░░░████████ 높음
     처음                              끝
          ↓        중간        ↓
          낮은 주의력 영역
          (여기 정보가 무시될 수 있음)

이 현상을 활용한 정보 배치 전략:

위치배치할 내용이유
처음 (시스템 프롬프트 직후)가장 중요한 규칙, 핵심 요구사항가장 높은 주의력 영역
중간참고 자료, 배경 정보, 예시보조 정보에 적합
끝 (사용자 입력 바로 앞)최종 지시사항, 출력 형식두 번째로 높은 주의력 영역

## 프로젝트 요약

- [기술 스택]: React, TypeScript, Tailwind CSS, Supabase
- [완료된 기능]: 로그인, 책 등록, 독서 메모

- 대시보드 컴포넌트 생성 완료

- 데이터 연결 미완료

표시할 차트:
1. 월별 독서량 (막대 차트)

실전 팁

"3줄 규칙": 프롬프트의 처음 3줄에 가장 중요한 요구사항을 적으세요. AI는 처음 부분에 가장 많은 주의를 기울입니다.

  • 마크다운 구조를 활용하세요: ## 헤딩, 번호 목록, 표를 사용하면 AI가 구조를 더 잘 파악합니다
  • 중복 제거: 같은 내용을 다른 표현으로 반복하면 토큰만 낭비됩니다
  • 대화 10턴 이상이면 요약 프롬프트: "지금까지 대화를 핵심 3줄로 요약해줘"를 활용
  • 불필요한 인사말, 감사 표현 제거: 정중함은 좋지만 토큰은 소중합니다. "안녕하세요, 제가 부탁이 있는데요..." 대신 바로 본론으로

주의사항

  • 컨텍스트 윈도우 크기를 과신하지 마세요: 200K 토큰이라고 해서 200K를 모두 채우면 성능이 떨어집니다. 70% 이내로 유지하는 것이 좋습니다.
  • 이전 대화를 무작정 복사하지 마세요: 필요한 정보만 요약해서 전달하는 것이 효과적입니다.
  • 한국어 토큰 비효율에 주의하세요: 한국어는 영어 대비 2~3배 토큰을 소비합니다. 시스템 프롬프트처럼 반복 사용되는 부분은 영어로 작성하는 것도 방법입니다.
  • 파일 전체를 붙여넣지 마세요: 관련 있는 부분만 발췌해서 전달하면 더 정확한 결과를 얻을 수 있습니다.

학습 목표

핵심 개념

시스템 프롬프트란?

AI에게 "너는 누구이고, 어떻게 행동해야 하는지"를 미리 알려주는 지시서입니다. 채팅에서 사용자가 매번 말하지 않아도 AI가 일관되게 행동하도록 설정합니다.

┌─────────────────────────────────────────────────┐
│              시스템 프롬프트 5대 구성요소            │
│                                                   │
│  ① 페르소나    — "너는 ~이다" (역할 정의)           │
│  ② 목표       — "~을 해야 한다" (핵심 임무)         │
│  ③ 규칙       — "~하지 마라" (제약 조건)            │
│  ④ 출력 형식   — "~형태로 답하라" (포맷 지정)        │
│  ⑤ 예시       — "이런 식으로" (참고 사례)            │
└─────────────────────────────────────────────────┘

페르소나 설정의 효과

같은 질문이라도 페르소나에 따라 응답의 깊이와 방향이 달라집니다.

페르소나 없이페르소나 있음
"버튼을 만들어줘" → 기본적인 버튼"너는 10년 경력의 UI/UX 디자이너야. 버튼을 만들어줘" → 접근성, 인터랙션, 시각적 계층 구조까지 고려한 버튼
일반적인 응답전문가 관점의 상세한 응답
표면적 결과맥락과 근거가 포함된 결과

너는 5년 경력의 풀스택 웹 개발자이자 교육 앱 전문가이다.
React, TypeScript, Tailwind CSS에 능숙하다.

사용자의 요청을 기반으로 교육용 웹 앱을 개발한다.

실습 3: 바이브코딩 도구별 시스템 프롬프트 커스터마이징

## 컨텍스트
Lovable은 자연어로 React 앱을 생성하는 바이브코딩 도구입니다.
shadcn/ui, Tailwind CSS, Supabase를 기본 지원합니다.

## 프롬프트 최적화 규칙
1. UI 컴포넌트는 shadcn/ui 기준으로 요청 (Button, Card, Dialog 등)
2. 스타일링은 Tailwind 클래스명으로 구체적으로 지시
3. 데이터 연동이 필요하면 Supabase 스키마를 명시
4. 한 번에 하나의 기능에 집중 (멀티태스킹 지양)
5. 수정 요청 시 변경할 부분만 구체적으로 지시

## 피해야 할 것
- "예쁘게 만들어줘" (모호한 디자인 요청)
- "알아서 해줘" (의사결정을 AI에게 위임)
- 한 번에 5개 이상 기능 동시 요청

Cursor용 시스템 프롬프트 프레임

## 컨텍스트
Cursor는 코드 에디터 기반 AI 코딩 도구입니다.
기존 코드베이스를 이해하고 수정/추가하는 데 특화되어 있습니다.

## 프롬프트 최적화 규칙
1. @파일명으로 관련 파일을 명시적으로 참조
2. 변경 범위를 "이 함수만", "이 컴포넌트만"으로 한정
3. 기존 코드 패턴과 일관성 유지를 지시
4. 타입 안전성을 항상 요구 (TypeScript)
5. 변경 후 영향받는 파일 목록을 요청

## 피해야 할 것
- 프로젝트 전체를 한 번에 리팩토링 요청
- 컨텍스트 없이 "이거 고쳐줘" (어떤 파일의 어떤 부분인지 명시)
- 다른 언어/프레임워크로 전환 요청 (점진적으로 마이그레이션)

실전 팁

시스템 프롬프트의 황금비율: 페르소나 20% + 규칙 40% + 출력 형식 30% + 예시 10%. 규칙이 가장 비중이 커야 일관된 결과를 얻을 수 있습니다.

  • 시스템 프롬프트는 한 번 잘 만들어 재사용하세요: 매번 새로 작성하는 것보다 검증된 프롬프트를 템플릿화
  • "하지 마라" 규칙이 효과적입니다: "간결하게 써줘"보다 "200자를 넘기지 마라"가 더 정확
  • 페르소나에 경력 연수를 넣으면 응답 깊이가 달라집니다: "개발자"보다 "10년 경력 시니어 개발자"
  • 출력 형식을 지정하면 후처리가 쉽습니다: 마크다운 표, JSON, 번호 목록 등 구체적으로

주의사항

  • 시스템 프롬프트가 너무 길면 역효과: 규칙이 20개 이상이면 AI가 일부를 무시할 수 있습니다. 핵심 규칙 10개 이내로 유지하세요.
  • 모순되는 규칙 주의: "간결하게 써줘"와 "모든 내용을 상세히 설명해줘"는 충돌합니다.
  • 페르소나 남용 주의: "너는 세계 최고의 천재 개발자야"보다 "너는 React와 TypeScript 전문 시니어 개발자야"가 더 효과적입니다.
  • 시스템 프롬프트만으로 해결하려 하지 마세요: 시스템 프롬프트는 "기본 설정"이고, 각 요청에서 세부사항을 추가하는 것이 좋습니다.

┌─────────────────────────────────────────────────────┐
│              프롬프트 실패 원인 5대 유형                │
│                                                      │

│  ④ 형식 미지정 — 출력 형태를 AI가 임의 결정           │

└─────────────────────────────────────────────────────┘

증상원인해결법

결과 형태가 원하는 것과 다르다형식 미지정출력 형식을 예시와 함께 지정

실습 1: Lovable 최적 프롬프트 패턴

Lovable은 "기획서 스타일" 프롬프트에 가장 잘 반응합니다.

패턴: 기능 명세 + UI 설명 + 데이터 구조

## 프로젝트: 학급 출석 체크 앱

### 페이지 구성
1. 로그인 페이지
2. 메인 대시보드
3. 출석 체크 페이지
4. 통계 페이지

### 메인 대시보드 상세
- 상단: 오늘 날짜와 요일 표시
- 중앙: 출석 현황 카드 3개 (출석/결석/지각 수)
- 하단: 최근 5일 출석률 미니 차트

### 출석 체크 페이지 상세
- 학생 목록을 카드 형태로 표시 (학번, 이름, 사진 아이콘)
- 각 학생 카드에 출석/결석/지각/조퇴 버튼 4개
- 기본값: 출석 (초록색 활성화)
- 전체 출석 체크 버튼 (한 번에 전원 출석 처리)

### 기술 요구사항
- React + TypeScript + Tailwind CSS
- 데이터: Supabase
- 테이블: students (id, name, class_id), attendance (id, student_id, date, status)

Lovable에서 효과적인 수정 프롬프트:

출석 체크 페이지에서:
1. 학생 카드의 높이를 줄여줘 (현재 너무 큼)
2. 출석 버튼 색상 변경: 출석=초록, 결석=빨강, 지각=노랑, 조퇴=회색
3. 전체 출석 버튼을 페이지 상단으로 이동

나머지는 그대로 유지해줘.

Cursor는 "파일 참조 + 구체적 지시" 스타일에 최적화되어 있습니다.

패턴: 파일 참조(@) + 변경 범위 한정 + 기존 패턴 유지

@components/AttendanceCard.tsx
@types/student.ts
@hooks/useAttendance.ts

AttendanceCard 컴포넌트를 수정해줘:

1. 현재 단일 선택 방식을 멀티 선택으로 변경
   - 여러 학생을 체크박스로 선택 후 일괄 처리
   - 선택된 학생 수를 상단에 표시

2. useAttendance 훅에 bulkUpdate 함수 추가
   - 선택된 학생 ID 배열과 상태를 받아 일괄 업데이트
   - 기존 updateAttendance 함수의 패턴을 따라 작성

3. student.ts의 타입에 isSelected?: boolean 추가

기존 UI 스타일과 일관성을 유지해줘.

- 예상 동작: 클릭 시 해당 학생의 출석 상태가 토글됨

실습 4: AntiGravity 최적 프롬프트 패턴

AntiGravity는 "아이디어 중심 + 빠른 프로토타이핑" 스타일에 적합합니다.

패턴: 핵심 아이디어 + 주요 화면 설명

학교 급식 메뉴 안내 웹앱을 만들어줘.

핵심 기능:
1. 이번 주 급식 메뉴 표시 (월~금)
2. 알레르기 정보 표시 (아이콘으로)
3. 메뉴 평가 (좋아요/별로예요)
4. 내일 메뉴 미리보기 알림

디자인:
- 귀여운 일러스트 느낌
- 파스텔 색상 (연보라, 연분홍, 연하늘)
- 각 메뉴에 음식 이모지 자동 매칭

실전 팁

도구 선택 기준: 새로 만들 때 = Lovable/AntiGravity, 기존 코드 수정 = Cursor, 복잡한 멀티파일 작업 = Claude Code

  • Lovable은 첫 프롬프트가 중요합니다: 초기 구조를 잘 잡아야 이후 수정이 쉽습니다
  • Cursor는 @(파일 참조)를 적극 활용하세요: AI가 관련 코드를 정확히 이해합니다
  • Claude Code는 터미널 명령도 함께 요청할 수 있습니다: "패키지 설치 후 코드 작성"을 한 번에
  • 도구를 바꿔도 괜찮습니다: Lovable로 초기 프로토타입을 만들고, Cursor로 세밀한 수정을 하는 것이 효과적인 워크플로입니다

주의사항

  • 도구의 한계를 인식하세요: Lovable은 복잡한 백엔드 로직에 약하고, Cursor는 처음부터 앱을 만드는 데 비효율적입니다.
  • 도구 전환 시 프롬프트도 바꾸세요: Lovable 스타일 프롬프트를 Cursor에 그대로 쓰면 비효율적입니다.
  • 하나의 도구에 종속되지 마세요: 각 도구의 강점을 조합하여 사용하는 것이 가장 효과적입니다.
  • 도구 업데이트에 주의하세요: 바이브코딩 도구는 빠르게 발전하므로 이전에 효과적이었던 프롬프트가 변경될 수 있습니다.

6. Few-shot과 Chain-of-Thought 활용

학습 목표

  • Few-shot 프롬프팅으로 AI에게 예시 기반 학습을 시키는 방법을 익힌다
  • Chain-of-Thought(CoT)로 단계적 사고를 유도하는 기법을 배운다
  • 자기 검증 프롬프트로 AI의 결과 품질을 높이는 방법을 실습한다

핵심 개념

Few-shot 프롬프팅이란?

AI에게 "이런 식으로 해줘" 라는 예시를 몇 개 보여주고, 동일한 패턴으로 새로운 결과를 생성하도록 하는 기법입니다.

┌─────────────────────────────────────────────────────┐
│               Few-shot 프롬프팅 구조                   │
│                                                       │
│  [지시사항]                                           │
│  "다음 형식으로 컴포넌트 코드를 생성해줘"               │
│                                                       │
│  [예시 1] ← 1-shot                                   │
│  입력: "로그인 버튼"                                   │
│  출력: (예시 코드)                                     │
│                                                       │
│  [예시 2] ← 2-shot                                   │
│  입력: "알림 배지"                                     │
│  출력: (예시 코드)                                     │
│                                                       │
│  [실제 요청] ← AI가 패턴을 학습하여 답변               │
│  입력: "프로필 카드"                                   │
│  출력: (AI가 패턴에 맞게 생성)                          │
└─────────────────────────────────────────────────────┘

Few-shot 유형 비교

유형예시 수적합한 상황토큰 비용
Zero-shot0개간단한 작업, 모델이 잘 아는 패턴최소
One-shot1개출력 형식을 지정할 때낮음
Few-shot (2~5개)2~5개복잡한 패턴, 일관성이 중요할 때중간
Many-shot (6개 이상)6개+매우 특수한 패턴높음

실전 권장: 대부분의 바이브코딩 작업은 2~3개 예시(Few-shot)로 충분합니다. 예시가 너무 많으면 오히려 토큰을 낭비하고 핵심이 흐려질 수 있습니다.

Chain-of-Thought(CoT)란?

AI에게 "결론만 말하지 말고, 생각 과정을 단계별로 보여줘" 라고 요청하는 기법입니다. 복잡한 문제에서 더 정확한 결과를 얻을 수 있습니다.

┌─────────────────────────────────────────────────────┐
│              CoT 없이 vs CoT 사용                     │
│                                                       │
│  Without CoT:                                         │
│  "이 코드의 버그를 찾아줘" → "line 15에 버그 있음"     │
│  (과정이 보이지 않아 신뢰하기 어려움)                   │
│                                                       │
│  With CoT:                                            │
│  "이 코드의 버그를 단계별로 분석해줘"                   │
│  → 1단계: 코드 흐름 분석                               │
│  → 2단계: 변수 타입 확인                               │
│  → 3단계: 경계 조건 점검                               │
│  → 4단계: line 15에서 null 참조 발견                   │
│  → 결론: null 체크 추가 필요                           │
│  (과정이 보여 검증 가능)                                │
└─────────────────────────────────────────────────────┘

실습 2: Few-shot으로 코드 스타일 통일

목표: 프로젝트의 코딩 컨벤션을 AI에게 학습시키기

우리 프로젝트의 React 커스텀 훅 작성 스타일입니다.
아래 예시를 참고하여 새로운 훅을 작성해줘.

### 예시 1: useLocalStorage

// hooks/useLocalStorage.ts
import { useState, useCallback } from 'react';

interface UseLocalStorageReturn<T> {
  value: T;
  setValue: (newValue: T) => void;
  removeValue: () => void;
}

export function useLocalStorage<T>(
  key: string,
  initialValue: T
): UseLocalStorageReturn<T> {
  // 초기값 로드
  const [value, setStoredValue] = useState<T>(() => {
    try {
      const item = window.localStorage.getItem(key);
      return item ? JSON.parse(item) : initialValue;
    } catch {
      return initialValue;
    }
  });

  // 값 설정
  const setValue = useCallback((newValue: T) => {
    setStoredValue(newValue);
    window.localStorage.setItem(key, JSON.stringify(newValue));
  }, [key]);

  // 값 제거
  const removeValue = useCallback(() => {
    setStoredValue(initialValue);
    window.localStorage.removeItem(key);
  }, [key, initialValue]);

  return { value, setValue, removeValue };
}

### 스타일 규칙 (예시에서 추출)
1. 파일 위치: hooks/ 디렉토리
2. 반환 타입을 인터페이스로 명시
3. 제네릭 사용
4. useCallback으로 함수 래핑
5. try-catch로 에러 처리
6. 한국어 주석 (// 형식)

### 요청
위 스타일을 따라 useDebounce 훅을 작성해줘.
- 입력값과 딜레이(ms)를 받아 디바운스된 값을 반환

실습 3: Chain-of-Thought로 복잡한 문제 해결

목표: 코드 아키텍처 설계에 CoT 적용

학급 관리 앱의 상태 관리 아키텍처를 설계해줘.

단계별로 사고 과정을 보여줘:

1단계: 관리해야 할 상태 목록을 나열해줘
   - 어떤 데이터가 전역 상태여야 하는지
   - 어떤 데이터가 로컬 상태여도 되는지

2단계: 상태 관리 도구를 비교해줘
   - Context API vs Zustand vs Jotai
   - 각각의 장단점을 이 프로젝트 기준으로 평가

3단계: 선택한 도구로 상태 구조를 설계해줘
   - 스토어 분리 기준
   - 타입 정의
   - 액션/셀렉터 설계

4단계: 최종 설계를 검증해줘
   - 이 설계로 해결할 수 없는 시나리오가 있는지
   - 성능 병목이 될 수 있는 부분은 없는지

기능 요구사항:
- 학생 목록 (30명)
- 출석 데이터 (일별)
- 좌석 배치 (드래그앤드롭)
- 사용자 로그인 상태

실습 4: 자기 검증 프롬프트

AI가 스스로 결과를 검토하도록 유도하는 기법입니다.

아래 요구사항으로 React 컴포넌트를 작성해줘.

[요구사항]
- 학생 출석부 테이블
- 이름, 출석 상태, 메모 컬럼
- 출석 상태: 드롭다운 (출석/결석/지각/조퇴)
- 메모: 인라인 편집 가능

[작성 후 자체 검증]
코드 작성이 끝나면 아래 체크리스트로 스스로 검증해줘:
- [ ] TypeScript 타입이 모두 정의되어 있는가?
- [ ] 접근성(aria) 속성이 포함되어 있는가?
- [ ] 에러 처리가 되어 있는가?
- [ ] 빈 데이터 상태(empty state)를 처리했는가?
- [ ] 모바일에서도 사용 가능한가?

검증 결과 미흡한 항목이 있으면 코드를 수정해서 다시 보여줘.

실전 팁

Few-shot의 핵심: 예시의 "수"보다 "질"이 중요합니다. 대표적이고 다양한 예시 2~3개가 비슷한 예시 10개보다 효과적입니다.

  • 예시에 "안티패턴"도 포함하세요: "이런 식으로는 하지 마" 예시를 함께 보여주면 AI가 더 정확히 이해합니다
  • CoT는 복잡한 문제에만 사용하세요: 단순한 작업에 CoT를 적용하면 불필요하게 긴 응답이 나옵니다
  • 자기 검증 프롬프트를 습관화하세요: "작성 후 ~를 확인해줘"만 추가하면 결과 품질이 크게 향상됩니다
  • Few-shot 예시를 라이브러리로 모아두세요: 프로젝트별, 패턴별로 정리해두면 재사용이 쉽습니다

주의사항

  • 예시가 잘못되면 결과도 잘못됩니다: Few-shot의 예시는 "정답"이어야 합니다. 잘못된 예시를 주면 AI가 잘못된 패턴을 학습합니다.
  • CoT 과다 사용 주의: 단순한 버튼 색상 변경에 "단계별로 사고해줘"는 과합니다.
  • 예시의 복잡도가 실제 요청과 비슷해야 합니다: 예시는 간단한데 실제 요청은 복잡하면 패턴 매칭이 안 됩니다.
  • 자기 검증 체크리스트가 너무 길면 역효과: 5~7개 항목이 적정합니다. 너무 많으면 핵심이 흐려집니다.

핵심 개념

왜 프롬프트 템플릿이 필요한가?

매번 프롬프트를 처음부터 작성하는 것은 비효율적입니다. 자주 사용하는 프롬프트를 템플릿화하면:

매번 새로 작성템플릿 사용
작성에 5~10분 소요변수만 바꾸면 1분 이내
품질이 들쭉날쭉일관된 품질 보장
좋은 프롬프트를 잊어버림검증된 프롬프트 재사용
개인에게만 유용팀원과 공유 가능

관리 규칙:

  1. 새로운 프롬프트가 3회 이상 재사용되면 템플릿으로 등록
  2. 각 템플릿에 "마지막 사용일"과 "효과 평가(1~5)"를 기록
  3. 분기별로 사용하지 않는 템플릿을 정리(아카이브)
  4. 팀원과 공유할 때는 변수 부분에 작성 가이드를 포함

실전 팁

좋은 템플릿의 기준: 변수 부분만 바꾸면 5분 이내에 프롬프트가 완성되어야 합니다. 변수를 채우는 데 10분 이상 걸린다면 템플릿이 너무 복잡한 것입니다.

  • 메모 앱에 템플릿을 저장하세요: Notion, Obsidian, 메모장 등 빠르게 복사-붙여넣기 할 수 있는 곳
  • "상황별 베스트 프롬프트"를 꾸준히 업데이트하세요: 더 좋은 결과를 얻은 프롬프트로 교체
  • 변수명은 직관적으로: {{x}}보다 {{프로젝트명}}이 사용할 때 실수가 적습니다
  • 템플릿 + 체이닝 조합: 각 체이닝 단계에 템플릿을 적용하면 일관된 고품질 결과를 얻을 수 있습니다

주의사항

  • 템플릿을 맹목적으로 따르지 마세요: 템플릿은 출발점이지 정답이 아닙니다. 상황에 따라 유연하게 수정하세요.
  • 과도한 변수화 주의: 변수가 10개 이상이면 오히려 사용이 번거롭습니다. 핵심 변수 5~7개가 적정합니다.
  • 오래된 템플릿 주의: AI 모델과 바이브코딩 도구는 빠르게 변합니다. 3개월 이상 업데이트하지 않은 템플릿은 검토가 필요합니다.
  • 템플릿 수집에 시간을 쓰지 마세요: 10개의 범용 템플릿이 100개의 특수 템플릿보다 유용합니다.
