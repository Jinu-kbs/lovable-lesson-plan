# 하네스 엔지니어링 — AI 시대의 실행 환경 설계

> 바이브코더 심화 가이드 · 2026년 3월

---

## 1. 하네스(Harness)란 무엇인가

### 비유로 이해하기

자동차 안에는 수백 개의 전선이 있습니다. 배터리에서 엔진으로, 엔진에서 전조등으로, 에어백으로, 오디오 시스템으로. 이 수백 개의 전선이 뒤엉켜 있다면 고장이 났을 때 어떤 전선이 문제인지 알 수가 없습니다.

자동차 공장에서는 이 문제를 **와이어링 하네스(Wiring Harness)**로 해결합니다. 전선들을 묶음으로 정리하고, 각 묶음에 역할을 부여하고, 표준 커넥터로 연결합니다. 복잡한 시스템을 **제어 가능한 구조**로 만드는 것입니다.

소프트웨어에서 **하네스(Harness)**도 똑같은 역할을 합니다.

```
[복잡한 실행 환경]
   코드 + 테스트 + AI 에이전트 + 자동화 도구
            ↓ 하네스가 정리
   [제어 가능한 구조]
   명확한 역할 + 자동 실행 + 오류 차단
```

### 하네스의 3가지 맥락

| 맥락 | 이름 | 역할 |
|------|------|------|
| 코드 검증 | 테스트 하네스 | 소프트웨어 컴포넌트를 자동으로 실행·검증 |
| AI 에이전트 | 에이전트 하네스 | AI 모델이 올바르게 작동하도록 환경 설계 |
| 자동화 시스템 | 훅(Hooks) 하네스 | 특정 시점에 자동으로 규칙을 실행 |

## 2. 테스트 하네스 (Test Harness)

### 핵심 개념

테스트 하네스는 소프트웨어 컴포넌트를 **독립적으로 실행하고 검증**할 수 있는 자동화 환경입니다.

구성요소 2가지:

**드라이버(Driver)**: 테스트할 컴포넌트를 호출하는 존재
```
드라이버가 하는 일:
- "이 함수를 이 입력값으로 실행해봐"
- 결과를 받아서 예상값과 비교
- 합격/불합격 판정
```

**스텁(Stub)**: 의존성을 가짜로 대체하는 존재
```
예시: 결제 모듈을 테스트하는데
실제 카드사 API를 쓰면 매번 돈이 빠져나간다.
스텁은 "결제 성공했어요"라고 답하는 가짜 카드사 역할
```

### 바이브코딩에서의 테스트 하네스

Lovable에서 만든 컴포넌트가 올바르게 작동하는지 확인하는 환경을 생각해보세요.

```
[테스트 하네스 없이]
- 코드 수정 → 브라우저 열기 → 직접 클릭해서 확인
- 수정할 때마다 반복 (매우 느림)

[테스트 하네스 있을 때]
- 코드 수정 → 자동으로 100개 시나리오 검증 → 결과 즉시 확인
```

### 중요한 개념: 테스트 피라미드

```
          /\
         /E2E\        ← 통합 테스트 (실제 브라우저로 전체 흐름 확인)
        /------\
       /통합 테스트\   ← 여러 모듈이 함께 작동하는지 확인
      /----------\
     /  단위 테스트  \  ← 개별 함수/컴포넌트 하나하나 확인
    /--------------\
```

하네스는 이 피라미드 전체를 **자동화**합니다.

## 3. AI 에이전트 하네스

### 에이전트와 하네스의 관계

Claude Code, Cursor AI, 안티그래비티 같은 AI 코딩 도구들은 단순한 "코드 완성기"가 아닙니다. 이들은 **에이전트(Agent)** — 스스로 계획을 세우고, 도구를 사용하고, 결과를 검증하는 자율적 실행자입니다.

그런데 에이전트가 제멋대로 움직이면 어떻게 될까요?

```
[하네스 없는 AI 에이전트]
개발자: "이 버그 고쳐줘"
에이전트: "네! 버그 고쳤어요. 그리고 이 파일도 좋아 보여서 리팩토링하고,
           저 파일도 삭제했고, 새 기능도 추가했어요!" 😱
```

**에이전트 하네스**는 AI가 올바른 범위 안에서 올바른 방식으로 작동하도록 환경을 설계합니다.

### 에이전트 하네스의 4가지 관리 요소

#### 1) Context (맥락 관리)
AI가 무엇을 알고 있는지 제어합니다.

```
CLAUDE.md 파일이 하는 일:
- 프로젝트 규칙 정의 ("개인정보 포함 인용 금지")
- 파일 구조 안내 ("이 파일은 이런 역할이야")
- 코딩 스타일 기준 ("CSS 변수는 이렇게 써")
```

#### 2) Tools (도구 접근 제어)
AI가 어떤 도구를 쓸 수 있는지 제어합니다.

```
허용 도구 예시:
✅ 파일 읽기, 파일 쓰기
✅ Git 커밋, 푸시
✅ 테스트 실행

차단 도구 예시:
❌ 시스템 파일 삭제
❌ 외부 API 무단 호출
```

#### 3) Memory (메모리 관리)
AI가 이전 대화를 어떻게 기억하는지 제어합니다.

```
프로젝트 메모리 (.claude/memory/)
- user.md: 사용자 선호도
- feedback.md: 이전에 잘 작동한/실패한 접근법
- project.md: 현재 진행 상황
```

#### 4) Lifecycle (생명주기 제어)
AI 작업의 시작·진행·종료를 제어합니다.

```
시작 → [맥락 로딩] → [도구 접근 설정] → 실행 → [검증] → 종료
                                                    ↕
                                              오류 시 롤백
```

### 실제 예시: Claude Code의 에이전트 하네스

```
사용자: "로그인 기능 만들어줘"

에이전트 하네스 동작:
1. CLAUDE.md 읽어서 프로젝트 규칙 확인
2. 기존 코드 구조 파악
3. 계획 수립 (단계별 TodoWrite)
4. 단계별 실행 (파일 읽기 → 수정 → 검증)
5. 결과 보고

규칙 위반 시:
→ 사용자에게 확인 요청
→ 허용하지 않은 작업은 차단
```

## 4. Claude Code Hooks 심화

### Hooks란 무엇인가

Hooks(훅)는 **특정 이벤트가 발생했을 때 자동으로 실행되는 스크립트**입니다.

중요한 포인트: 훅은 **AI 모델이 실행하는 것이 아닙니다.** Claude Code 하네스(실행 환경)가 직접 실행합니다. AI가 무시하거나 우회할 수 없는 **강제 규칙**입니다.

```
일반 규칙 (CLAUDE.md에 작성):
AI가 읽고 따르기를 "요청"하는 것
→ AI가 까먹거나 다른 판단을 할 수 있음

훅 (settings.json에 작성):
이벤트 발생 시 시스템이 강제 실행
→ AI가 절대 우회 불가
```

### 4가지 훅 유형

#### PreToolUse — 도구 사용 전 실행
```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "echo '⚠️ 쉘 명령어 실행 전 보안 검사'"
      }]
    }]
  }
}
```

언제 쓰나: 위험한 명령 실행 전 경고, 입력 검증, 로깅

#### PostToolUse — 도구 사용 후 실행
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit",
      "hooks": [{
        "type": "command",
        "command": "prettier --write ."
      }]
    }]
  }
}
```

언제 쓰나: 파일 수정 후 자동 포맷, 테스트 자동 실행, 변경 로깅

#### PreCompact — 컨텍스트 압축 전 실행
```json
{
  "hooks": {
    "PreCompact": [{
      "hooks": [{
        "type": "command",
        "command": "cat CLAUDE.md"
      }]
    }]
  }
}
```

언제 쓰나: 긴 대화에서 중요 정보 보존, 핵심 컨텍스트 재주입

#### Stop — 대화 종료 시 실행
```json
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "type": "command",
        "command": "echo '작업 완료. 결과를 검토해주세요.'"
      }]
    }]
  }
}
```

언제 쓰나: 작업 완료 알림, 정리 작업, 로그 저장

### 훅 설정 위치

```
~/.claude/settings.json    ← 전체 프로젝트에 적용 (글로벌)
.claude/settings.json      ← 이 프로젝트에만 적용 (로컬)
.claude/settings.local.json ← 로컬 전용 (Git에 커밋 안 함)
```

### 실전 예시: 코드 품질 자동화 하네스

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix $(git diff --name-only --diff-filter=M | grep '\\.js$')"
          },
          {
            "type": "command",
            "command": "npx prettier --write $(git diff --name-only --diff-filter=M)"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "npm test -- --passWithNoTests 2>&1 | tail -5"
          }
        ]
      }
    ]
  }
}
```

이 설정으로 무슨 일이 일어나나:
1. 파일 수정할 때마다 → ESLint로 코드 스타일 자동 교정
2. 파일 수정할 때마다 → Prettier로 포맷 자동 정렬
3. 작업 완료할 때마다 → 테스트 자동 실행 후 결과 표시

## 5. 실전: 바이브코더 하네스 설계

### 하네스의 3층 구조

```
┌─────────────────────────────────────────┐
│  Layer 3: Hooks (자동 실행 규칙)         │
│  settings.json                           │
│  - 코드 저장 시 자동 포맷                │
│  - 커밋 전 자동 테스트                   │
├─────────────────────────────────────────┤
│  Layer 2: Context (맥락 정의)            │
│  CLAUDE.md + .claude/memory/             │
│  - 프로젝트 규칙                         │
│  - 코딩 스타일                           │
│  - 사용자 선호도                         │
├─────────────────────────────────────────┤
│  Layer 1: Skills & Tools (능력 정의)    │
│  .claude/commands/ + MCP 서버            │
│  - 반복 작업 슬래시 명령어               │
│  - 외부 서비스 연동                      │
└─────────────────────────────────────────┘
```

### 초보자를 위한 최소 하네스 설계

처음부터 복잡하게 만들 필요 없습니다. 3가지만 설정해도 강력한 하네스입니다.

**Step 1: CLAUDE.md 작성 (맥락 레이어)**
```markdown
# 프로젝트 규칙

## 코딩 스타일
- 한국어 변수명 사용 금지
- 함수는 하나의 역할만

## 금지 사항
- console.log 남기지 않기
- 미사용 변수 남기지 않기

## 선호 패턴
- async/await (Promise.then 대신)
- CSS 변수 활용
```

**Step 2: 간단한 PostToolUse 훅 설정**
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit",
      "hooks": [{
        "type": "command",
        "command": "echo '파일 수정됨: 검토 필요'"
      }]
    }]
  }
}
```

**Step 3: /commit 슬래시 명령어 활용**
매번 커밋 메시지 쓰는 대신, Claude Code에 내장된 `/commit` 명령어로 자동화.

### 실전 하네스 예시: 교육용 코드 검토 하네스

```markdown
# CLAUDE.md — 학생 코딩 환경 하네스

## 역할
당신은 고등학생의 코딩 프로젝트를 돕는 AI 코딩 멘토입니다.

## 규칙
1. 코드를 직접 써주지 말고, 방향을 제시하세요
2. 에러 메시지는 해석해주되 해결책은 힌트만
3. 잘 작동하는 코드에는 칭찬 먼저
4. 코드 설명은 비유를 활용하세요

## 금지
- 완성된 코드 직접 제공
- 복잡한 기술 용어 사용 (설명 없이)
```

## 6. 추가로 알아야 할 개념

### Context Engineering (맥락 엔지니어링)

하네스 엔지니어링은 더 큰 개념인 **맥락 엔지니어링**의 구조적 부분입니다.

```
맥락 엔지니어링 = "AI가 올바른 결과를 내도록 입력을 설계하는 기술"

구성 요소:
- 하네스 엔지니어링 (실행 환경 설계)
- 프롬프트 엔지니어링 (지시문 설계)
- RAG (검색 증강 생성)
- Few-shot 예시 설계
```

좋은 프롬프트 하나보다 잘 설계된 하네스가 훨씬 강력합니다. 프롬프트는 일회성이지만, 하네스는 모든 작업에 자동 적용됩니다.

### MCP (Model Context Protocol)

MCP는 AI 에이전트가 외부 서비스와 연동하는 표준 프로토콜입니다.

```
[MCP 없이]
AI → 코드 작성 → 사람이 직접 API 호출 → 결과 다시 AI에게 전달

[MCP 있을 때]
AI → MCP 서버 → Notion API / Slack API / GitHub API
                  결과 직접 수신 → 다음 작업 자동 수행
```

Claude Code에서 MCP 서버 추가:
```bash
claude mcp add notion-server npx @notion/mcp-server
```

### 오케스트레이션 vs 자동화

자주 혼동하는 두 개념:

| 구분 | 자동화 (Automation) | 오케스트레이션 (Orchestration) |
|------|--------------------|-----------------------------|
| 뜻 | 반복 작업을 기계가 대신 | 여러 자동화를 조율 |
| 예시 | 파일 저장 시 포맷 | 파일 저장 → 테스트 → 배포 → 알림 |
| 하네스 위치 | 개별 훅 | 전체 파이프라인 |

하네스 엔지니어링은 자동화와 오케스트레이션을 모두 다룹니다.

---

## 핵심 요약

```
하네스 엔지니어링
= 코드가 실행되는 환경을 설계하는 기술

3가지 핵심:
1. 테스트 하네스 → "내 코드가 올바른지 자동으로 확인"
2. 에이전트 하네스 → "AI가 올바르게 행동하도록 환경 설계"
3. Hooks 하네스 → "특정 시점에 자동으로 규칙 실행"

바이브코더에게 의미하는 바:
"코드를 잘 짜는 것"보다
"AI가 코드를 잘 짜도록 환경을 잘 설계하는 것"이
더 중요한 시대가 왔다.
```
