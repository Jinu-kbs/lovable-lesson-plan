## 1. 프롬프트 엔지니어링이란?

### 학습 목표

- 프롬프트의 개념과 프롬프트 엔지니어링이 왜 중요한지 이해한다
- 좋은 프롬프트와 나쁜 프롬프트의 차이를 구별할 수 있다
- AI와 효과적으로 대화하는 것이 "기술"임을 인식한다

### 프롬프트란 무엇인가요?

**프롬프트(Prompt)**는 AI에게 보내는 "요청" 또는 "지시"입니다. 쉽게 말해, AI에게 하는 질문이나 부탁이 바로 프롬프트입니다.

우리가 식당에서 주문할 때를 생각해볼까요?

| 상황 | 식당 주문 | AI 프롬프트 |
|------|----------|------------|
| 막연한 요청 | "뭐 맛있는 거 주세요" | "좋은 글 써줘" |
| 구체적 요청 | "매운 김치찌개 1인분, 밥 곱빼기로 주세요" | "중학생 대상, 300자 이내, 친근한 말투로 자기소개 글 써줘" |
| 결과 | 뭐가 나올지 모름 vs 원하는 메뉴 | 엉뚱한 답 vs 원하는 답 |

> 식당에서 구체적으로 주문할수록 원하는 음식이 나오듯, AI에게도 구체적으로 요청할수록 원하는 결과를 얻을 수 있습니다.

### 프롬프트 엔지니어링이 왜 중요한가요?

**프롬프트 엔지니어링(Prompt Engineering)**은 AI에게 효과적으로 지시하는 기술입니다.

같은 AI 모델을 사용해도 프롬프트에 따라 결과가 크게 달라집니다:

```
같은 AI + 좋은 프롬프트 = 훌륭한 결과
같은 AI + 나쁜 프롬프트 = 엉뚱한 결과

→ AI의 성능이 아니라 "질문하는 방법"이 결과를 결정합니다!
```

#### 비유: AI는 유능하지만 마음을 못 읽는 비서

AI를 매우 유능한 비서라고 생각해보세요:

- 지식이 풍부하고 일을 빠르게 처리합니다
- 하지만 당신의 마음을 읽지는 못합니다
- "알아서 해줘"라고 하면, 비서 나름의 판단으로 합니다
- "이런 식으로, 이 형식으로, 이 정도 분량으로 해줘"라고 하면, 정확히 원하는 결과가 나옵니다

프롬프트 엔지니어링은 이 유능한 비서에게 "명확한 업무 지시서"를 작성하는 기술입니다.

### Before/After: 좋은 질문 vs 나쁜 질문

#### 예시 1: 글쓰기

**Before (나쁜 프롬프트):**

```
자기소개 써줘
```

> 결과: AI가 아무 기준 없이 일반적인 자기소개를 생성. 누구의 자기소개인지, 어디에 쓸 건지, 어떤 톤인지 알 수 없음.

**After (좋은 프롬프트):**

```
나는 3년차 초등학교 교사입니다.
교육 블로그에 올릴 자기소개 글을 써줘.

조건:
- 분량: 200~300자
- 말투: 학부모가 읽어도 친근하면서 신뢰감 있는 톤
- 포함 내용: 교직 경력, 교육 철학(놀이 중심 학습), 블로그 운영 목적
- 제외: 개인정보(본명, 학교명)
```

> 결과: 구체적인 조건에 맞는, 바로 사용할 수 있는 자기소개가 생성됨.

---

#### 예시 2: 요약

**Before (나쁜 프롬프트):**

```
이 기사 요약해줘
```

> 결과: AI가 전체를 대충 줄여서 핵심이 빠질 수 있음.

**After (좋은 프롬프트):**

```
아래 기사를 다음 형식으로 요약해줘:

1. 핵심 주장 (1문장)
2. 근거 3가지 (각 1문장)
3. 반대 의견 (있다면)
4. 나의 활용 포인트 (교사로서 수업에 어떻게 활용할 수 있는지)

[기사 내용 붙여넣기]
```

> 결과: 체계적으로 정리된 요약이 나오며, 활용 방안까지 포함됨.

---

#### 예시 3: 아이디어 생성

**Before (나쁜 프롬프트):**

```
수업 아이디어 좀 줘
```

> 결과: 너무 일반적이거나 학년/과목에 맞지 않는 아이디어가 나옴.

**After (좋은 프롬프트):**

```
초등학교 5학년 과학 수업에서 "날씨와 우리 생활" 단원을 가르칩니다.

아래 조건에 맞는 수업 활동 아이디어 5가지를 제안해줘:
- 40분 수업 내 실행 가능
- 준비물이 적은 것 (교실에서 바로 가능)
- 학생 참여형 활동 (강의식 X)
- 각 아이디어마다: 활동명, 소요시간, 준비물, 진행방법 포함
```

> 결과: 바로 수업에 적용할 수 있는 구체적인 활동 아이디어가 나옴.

---

#### 예시 4: 번역

**Before (나쁜 프롬프트):**

```
이거 영어로 번역해줘
```

> 결과: 문맥에 맞지 않는 직역이 나올 수 있음.

**After (좋은 프롬프트):**

```
아래 한국어 이메일을 영어로 번역해줘.

조건:
- 비즈니스 이메일 톤 (formal but friendly)
- 수신자: 미국 파트너사 담당자
- 한국식 표현은 영어권에 자연스러운 표현으로 의역
- 번역 후 원문과 대조할 수 있도록 한영 병렬로 보여줘

[이메일 내용]
```

> 결과: 상황에 맞는 자연스러운 비즈니스 영어 이메일이 생성됨.

---

#### 예시 5: 코드 작성

**Before (나쁜 프롬프트):**

```
계산기 만들어줘
```

> 결과: 어떤 언어로, 어떤 종류의 계산기를 만들지 불분명. 콘솔 프로그램? 웹 앱? 과학 계산기?

**After (좋은 프롬프트):**

```
HTML/CSS/JavaScript로 웹 계산기를 만들어줘.

요구사항:
- 기본 사칙연산 (+, -, x, ÷)
- 화면: 숫자 버튼 0~9, 연산자 버튼, = 버튼, C(초기화) 버튼
- 디자인: 아이폰 계산기 스타일, 다크 테마
- 결과 표시: 상단에 큰 글씨로 숫자 표시
- 파일 1개(index.html)에 모든 코드 포함
```

> 결과: 바로 실행 가능한 웹 계산기 코드가 생성됨.

### 핵심 정리

| 구분 | 나쁜 프롬프트 | 좋은 프롬프트 |
|------|-------------|-------------|
| **구체성** | "~해줘" 한 줄 | 대상, 목적, 형식, 제약 조건 명시 |
| **맥락** | 상황 설명 없음 | 누가, 왜, 어디에 쓸 건지 설명 |
| **형식** | 결과 형태 미지정 | 원하는 출력 형식 지정 |
| **분량** | 분량 미지정 | 글자 수, 항목 수 등 명시 |
| **결과** | 운에 맡김 | 원하는 답을 높은 확률로 얻음 |

> **기억하세요**: 프롬프트 엔지니어링은 "AI를 잘 다루는 기술"이 아니라 "내가 원하는 것을 명확히 표현하는 기술"입니다. AI가 똑똑해질수록, 프롬프트를 잘 쓰는 사람과 못 쓰는 사람의 결과 차이는 더 커집니다.

## 2. 5가지 기본 원칙

### 학습 목표

- 효과적인 프롬프트의 5가지 기본 원칙을 이해한다
- 각 원칙을 적용한 Before/After 예시를 통해 실전 감각을 익힌다
- 5가지 원칙을 조합하여 프롬프트를 작성할 수 있다

### 원칙 한눈에 보기

```
┌─────────────────────────────────────────────────┐
│           프롬프트 5대 원칙                       │
│                                                  │
│  ① 구체적으로 말하기     ← 가장 기본이자 핵심      │
│  ② 역할 부여하기        ← AI의 전문성을 끌어내기   │
│  ③ 예시 제공하기         ← Few-shot 기법           │
│  ④ 단계적으로 요청하기   ← Chain-of-Thought        │
│  ⑤ 제약 조건 설정하기    ← 범위와 형식 제한         │
└─────────────────────────────────────────────────┘
```

---

### 원칙 1: 구체적으로 말하기 (Be Specific)

가장 기본이면서 가장 중요한 원칙입니다. 막연한 요청 대신 구체적인 조건을 명시합니다.

#### 구체적으로 만드는 체크리스트

프롬프트를 쓴 후 아래 항목을 점검하세요:

- [ ] **누구를 위한 건가요?** (대상/독자)
- [ ] **무엇을 원하나요?** (과제/목표)
- [ ] **어떤 형식이 좋은가요?** (표, 목록, 문단 등)
- [ ] **얼마나 필요한가요?** (분량, 개수)
- [ ] **어떤 톤이 좋은가요?** (격식, 친근, 전문적 등)

#### Before → After 비교

**Before:**

```
여행 계획 세워줘
```

**After:**

```
2박 3일 제주도 가족여행 계획을 세워줘.

조건:
- 여행자: 부부 + 초등학생 2명 (8세, 11세)
- 시기: 2026년 4월 (봄)
- 예산: 교통·숙박 제외 체험비 30만원 이내
- 선호: 자연 체험, 아이들 교육적 장소
- 비선호: 쇼핑몰, 실내 놀이공원
- 형식: 일자별 표 형식 (시간, 장소, 활동, 예상비용)
```

> **왜 좋은가요?** 대상(가족 구성), 시기, 예산, 취향, 출력 형식이 모두 명시되어 AI가 정확하게 계획을 세울 수 있습니다.

---

### 원칙 2: 역할 부여하기 (Role Assignment)

AI에게 특정 전문가 역할을 부여하면, 해당 분야의 전문 지식과 관점에서 답변합니다.

#### 역할 부여의 원리

```
일반 AI 모드:    "다방면의 지식을 가진 일반인"처럼 답변
역할 부여 모드:  "해당 분야 전문가"처럼 답변

→ 같은 질문이라도 전문적이고 깊이 있는 답변을 얻을 수 있음
```

#### 자주 쓰는 역할 모음

| 역할 | 프롬프트 시작 문구 | 활용 상황 |
|------|-------------------|----------|
| 교육 전문가 | "당신은 10년 경력의 초등교육 전문가입니다" | 수업 자료, 학습 활동 설계 |
| 카피라이터 | "당신은 SNS 마케팅 전문 카피라이터입니다" | 홍보 문구, 광고 카피 작성 |
| 번역가 | "당신은 한영 통번역 10년 경력의 번역가입니다" | 자연스러운 번역 |
| 개발자 | "당신은 시니어 프론트엔드 개발자입니다" | 코드 작성, 기술 조언 |
| 상담사 | "당신은 진로상담 전문 상담교사입니다" | 진로 조언, 상담 시나리오 |
| 기획자 | "당신은 IT 서비스 기획자입니다" | PRD, 기능 정의, 와이어프레임 |
| 편집자 | "당신은 출판사 편집자입니다" | 글 다듬기, 교정교열 |
| 데이터 분석가 | "당신은 교육 데이터 분석 전문가입니다" | 설문 분석, 통계 해석 |

#### Before → After 비교

**Before:**

```
이 글 좀 고쳐줘.

[글 내용]
```

**After:**

```
당신은 교육 전문 출판사에서 10년간 일한 편집자입니다.
아래 글을 다음 기준으로 교정해주세요:

교정 기준:
1. 문법 오류 수정
2. 문장이 너무 길면 2개로 나누기
3. 어려운 용어는 쉬운 말로 바꾸기 (독자: 학부모)
4. 수정한 부분은 【수정 전 → 수정 후】 형식으로 표시

[글 내용]
```

> **왜 좋은가요?** "편집자" 역할을 부여함으로써 AI가 전문적인 교정교열 관점에서 접근합니다. 구체적인 교정 기준까지 제시하여 일관된 결과를 얻을 수 있습니다.

#### 역할 부여 고급 팁: 페르소나 상세 설정

단순한 직업명보다 구체적인 페르소나를 설정하면 더 좋은 결과를 얻을 수 있습니다:

```
[기본] 당신은 교사입니다.

[고급] 당신은 서울에서 15년간 초등학교 5학년을 가르친 경험 많은 교사입니다.
특히 과학 교과에 전문성이 있으며, 실험 중심 수업을 선호합니다.
최근에는 AI를 활용한 수업 혁신에 관심이 많습니다.
학생들이 "선생님 수업은 항상 재밌어요!"라고 말할 정도로
학생 참여형 수업을 잘 설계합니다.
```

---

### 원칙 3: 예시 제공하기 (Few-shot Prompting)

AI에게 원하는 결과의 예시를 보여주면, 같은 패턴으로 결과를 생성합니다.

#### Few-shot의 원리

```
Zero-shot:  예시 없이 지시만 → AI가 알아서 형식 결정
One-shot:   예시 1개 제공    → AI가 패턴을 파악
Few-shot:   예시 2~3개 제공  → AI가 패턴을 정확히 학습

→ 예시가 많을수록 원하는 형식에 정확히 맞는 결과를 얻을 수 있음
```

#### Before → After 비교

**Before:**

```
아래 단어들의 뜻을 설명해줘:
API, 프론트엔드, 백엔드, 데이터베이스
```

> 결과: AI가 각각 다른 형식으로, 때로는 너무 전문적으로 설명함.

**After:**

```
아래 IT 용어들을 초보자가 이해할 수 있게 설명해줘.
다음 예시와 같은 형식으로 작성해줘:

[예시]
용어: 서버
한줄 설명: 웹사이트의 데이터를 저장하고 보내주는 컴퓨터
비유: 도서관의 사서 — 책(데이터)을 보관하고 있다가, 누가 요청하면 찾아서 건네줌
실생활 예: 네이버에 접속하면 네이버의 서버가 웹페이지를 당신의 브라우저로 보내줌

---

이 형식으로 설명할 용어:
1. API
2. 프론트엔드
3. 백엔드
4. 데이터베이스
```

> **왜 좋은가요?** 예시를 통해 "한줄 설명 + 비유 + 실생활 예"라는 형식을 정확히 보여줬으므로, AI가 동일한 구조로 나머지 용어도 설명합니다.

#### Few-shot 활용 팁

| 상황 | 권장 예시 수 | 이유 |
|------|------------|------|
| 간단한 형식 변환 | 1~2개 | 패턴이 단순하므로 적은 예시로 충분 |
| 특정 톤/스타일 지정 | 2~3개 | 톤을 일관되게 맞추려면 여러 예시 필요 |
| 복잡한 분류/판단 | 3~5개 | 다양한 케이스를 보여줘야 정확도 향상 |
| 창의적 생성 | 1~2개 | 너무 많으면 오히려 창의성 제한 |

---

### 원칙 4: 단계적으로 요청하기 (Chain-of-Thought)

복잡한 작업을 한 번에 요청하지 말고, 단계를 나누어 요청합니다.

#### 단계적 요청의 원리

```
한 번에 모든 것:  "레포트 써줘" → 구조가 불분명한 긴 글
단계적 요청:
  1단계: "목차를 먼저 잡아줘"
  2단계: "이 목차에서 1장을 써줘"
  3단계: "2장을 써줘, 1장의 톤을 유지해서"

→ 각 단계의 결과를 확인하고 방향을 조정할 수 있음
```

#### Before → After 비교

**Before:**

```
학교 소개 웹사이트 만들어줘
```

> 결과: AI가 한 번에 모든 것을 만들려다 보니, 구조가 뒤죽박죽이거나 중요한 섹션이 빠질 수 있음.

**After:**

```
학교 소개 웹사이트를 만들려고 합니다. 단계별로 진행해주세요.

Step 1: 먼저 웹사이트에 필요한 페이지 목록과 각 페이지의 주요 섹션을 정리해주세요.
(이 단계의 결과를 확인한 후 다음 단계를 요청하겠습니다)
```

> 이후 대화:

```
좋아요! 그 구조로 진행합니다.

Step 2: 메인 페이지(index.html)의 HTML 코드를 작성해주세요.
조건:
- 학교 이름: 행복초등학교
- 색상: 파란색 + 흰색 기반
- 반응형 디자인
```

```
Step 3: 방금 만든 메인 페이지에 교직원 소개 섹션을 추가해주세요.
- 카드 형식으로 4명
- 사진 자리는 placeholder로
```

> **왜 좋은가요?** 각 단계의 결과를 확인하고, 필요하면 수정한 뒤 다음 단계로 넘어갈 수 있습니다. 중간에 방향 수정이 가능합니다.

#### Chain-of-Thought (사고의 연쇄) 기법

AI에게 답만 말하지 말고 "생각 과정을 보여달라"고 하면 더 정확한 결과를 얻을 수 있습니다:

```
아래 문제를 풀어줘. 답만 말하지 말고, 풀이 과정을 단계별로 보여줘.

문제: 한 학급에 30명의 학생이 있습니다. 남학생이 여학생보다 4명 많습니다.
남학생과 여학생은 각각 몇 명인가요?

풀이 형식:
1단계: 문제에서 주어진 조건 정리
2단계: 식 세우기
3단계: 풀이
4단계: 검증
5단계: 최종 답
```

> **팁**: "단계별로 생각해줘(Think step by step)"라는 한 문장만 추가해도 AI의 추론 정확도가 크게 향상됩니다.

### 원칙 5: 제약 조건 설정하기 (Constraints)

AI의 응답 범위와 형식을 제한하여, 원하는 결과만 얻습니다.

#### 자주 쓰는 제약 조건 유형

| 제약 유형 | 예시 | 효과 |
|----------|------|------|
| **분량 제한** | "200자 이내로" / "5문장으로" | 너무 길거나 짧은 답변 방지 |
| **형식 지정** | "표 형식으로" / "번호 목록으로" | 일관된 출력 형식 확보 |
| **포함/제외** | "반드시 ~를 포함" / "~는 제외" | 필요한 내용만 포함 |
| **톤 지정** | "격식체로" / "반말로" / "유머러스하게" | 상황에 맞는 톤 확보 |
| **대상 지정** | "초등학생이 이해할 수 있게" | 적절한 난이도 조절 |
| **언어 제한** | "한국어로만" / "영어 병기" | 언어 혼용 방지 |
| **금지 사항** | "전문 용어 사용 금지" | 부적절한 표현 방지 |

#### Before → After 비교

**Before:**

```
마케팅 이메일 써줘
```

> 결과: 너무 길거나, 부적절한 톤이거나, 핵심 메시지가 불분명한 이메일.

**After:**

```
신규 온라인 강좌 홍보 이메일을 써줘.

제약 조건:
- 분량: 본문 150~200자 (모바일에서 스크롤 없이 읽을 수 있는 정도)
- 톤: 친근하지만 전문적 (20~30대 직장인 대상)
- 구조: 제목줄 → 인사 → 핵심 혜택 3줄 → CTA 버튼 문구
- 필수 포함: 50% 할인, 마감일(3/31), 수강 후기 1줄
- 사용 금지: "지금 바로!", "놓치지 마세요!" 같은 과장된 표현
- 이메일 제목줄: 3가지 후보를 제안해줘
```

> **왜 좋은가요?** 분량, 톤, 구조, 필수 요소, 금지 사항까지 명확히 지정하여 바로 사용할 수 있는 이메일이 생성됩니다.

### 5가지 원칙 통합 예시

모든 원칙을 하나의 프롬프트에 적용한 종합 예시:

```
[역할 부여 - 원칙 2]
당신은 초등학교 ICT 교육을 10년간 담당한 경험 많은 교사입니다.

[맥락 - 원칙 1: 구체적으로]
5학년 학생들에게 "인터넷 안전" 주제로 40분 수업을 설계하려 합니다.

[과제 - 원칙 4: 단계적 요청]
아래 단계로 수업 지도안을 작성해주세요:
1. 수업 목표 3가지
2. 도입(5분) - 동기유발 활동
3. 전개(30분) - 핵심 활동 2개
4. 정리(5분) - 형성평가

[예시 제공 - 원칙 3]
활동 형식 예시:
| 활동명 | 방법 | 준비물 | 시간 |
| 피싱 메일 찾기 | 실제 피싱 메일 예시를 보고 가짜 구별 | PPT 슬라이드 | 15분 |

[제약 조건 - 원칙 5]
- 특수 기자재 없이 교실에서 바로 실행 가능
- 모둠 활동 1개 이상 포함
- 학생 활동지(워크시트) 포함
- 평가 루브릭 포함
```

> 이렇게 5가지 원칙을 모두 적용하면, AI가 체계적이고 실용적인 수업 지도안을 생성합니다.

### 원칙 암기 팁

5가지 원칙을 쉽게 기억하는 방법:

```
"구역예단제" (구체-역할-예시-단계-제약)

또는 문장으로:
"구체적으로, 역할을 주고, 예시를 보여주고, 단계별로 요청하며, 제약을 건다"
```

## 3. 프롬프트 구조 템플릿

### 학습 목표

- 프롬프트의 기본 구조(프레임워크)를 이해한다
- 상황별 실전 템플릿 5종을 활용할 수 있다
- 자신만의 프롬프트 템플릿을 만들 수 있다

### 프롬프트 구조 프레임워크: RCOTF

효과적인 프롬프트는 5가지 요소로 구성됩니다:

```
┌──────────────────────────────────────────────┐
│           RCOTF 프레임워크                     │
│                                               │
│  R - Role (역할)      "당신은 ~입니다"         │
│  C - Context (맥락)   "상황은 ~입니다"         │
│  O - Objective (과제) "~을 해주세요"           │
│  T - Tone (톤)        "~한 말투로"             │
│  F - Format (형식)    "~형식으로 출력"          │
│                                               │
│  + Constraints (제약)  "단, ~하지 마세요"       │
└──────────────────────────────────────────────┘
```

#### 각 요소 상세 설명

| 요소 | 질문 | 예시 |
|------|------|------|
| **R (역할)** | AI가 어떤 전문가 역할을 해야 하나요? | "10년 경력 UX 디자이너" |
| **C (맥락)** | 어떤 상황인가요? 배경은? | "교육용 웹사이트를 만들고 있음" |
| **O (과제)** | 무엇을 해야 하나요? | "메인 페이지 와이어프레임 제안" |
| **T (톤)** | 어떤 톤으로 답해야 하나요? | "친근하지만 전문적으로" |
| **F (형식)** | 어떤 형태로 출력해야 하나요? | "표 + 설명 문단" |
| **+ 제약** | 하지 말아야 할 것은? 범위는? | "3가지 이내, 전문 용어 설명 포함" |

#### RCOTF 적용 예시

```
[R - 역할]
당신은 초등학교에서 10년간 국어를 가르친 경험 많은 교사입니다.

[C - 맥락]
3학년 학생들의 독서 감상문 쓰기 능력이 부족합니다.
특히 "느낀 점"을 구체적으로 쓰는 것을 어려워합니다.

[O - 과제]
독서 감상문 쓰기 지도용 워크시트를 만들어주세요.

[T - 톤]
학생들이 부담 없이 작성할 수 있도록 쉽고 친근한 말투로 작성해주세요.

[F - 형식]
- A4 1장 분량
- 단계별 질문 형식 (빈칸 채우기)
- 예시 문장 포함

[제약 조건]
- 어려운 한자어 사용 금지
- 자유롭게 쓸 수 있는 빈칸도 포함
- 색칠하거나 그림 그릴 수 있는 공간 제안
```

### 실전 템플릿 5종

아래 템플릿을 상황에 맞게 복사하여 사용하세요. 대괄호 `[  ]` 안의 내용을 자신의 상황으로 바꾸면 됩니다.

#### 템플릿 1: 글쓰기/콘텐츠 생성

```
당신은 [분야] 전문 작가입니다.

[대상]을 위한 [주제]에 대한 [글의 종류]를 작성해주세요.

조건:
- 분량: [글자 수 또는 문단 수]
- 톤: [격식/비격식, 톤 설명]
- 구조: [서론-본론-결론 / 번호 목록 / 기타]
- 필수 포함: [반드시 들어갈 내용]
- 사용 금지: [피해야 할 표현이나 내용]
```

**사용 예시:**

```
당신은 교육 분야 전문 작가입니다.

학부모를 위한 "겨울방학 독서 추천"에 대한 뉴스레터 글을 작성해주세요.

조건:
- 분량: 500자 내외
- 톤: 친근하고 실용적 (엄마/아빠가 읽는 느낌)
- 구조: 도입 → 추천 도서 5권 → 독서 활동 팁 → 마무리
- 필수 포함: 학년별 추천 (저학년/고학년 구분)
- 사용 금지: 교육학 전문 용어
```

#### 템플릿 2: 분석/평가

```
당신은 [분야] 분석 전문가입니다.

아래 [분석 대상]을 분석해주세요.

분석 기준:
1. [기준 1]
2. [기준 2]
3. [기준 3]

출력 형식:
- 각 기준별 점수 (/10)
- 강점 [N]가지
- 약점 [N]가지
- 개선 제안 [N]가지

[분석 대상 내용 붙여넣기]
```

**사용 예시:**

```
당신은 웹사이트 UX 분석 전문가입니다.

아래 웹사이트 URL의 메인 페이지를 분석해주세요.

분석 기준:
1. 첫 인상 (3초 내 핵심 메시지 전달 여부)
2. 네비게이션 직관성
3. 모바일 반응형 대응

출력 형식:
- 각 기준별 점수 (/10)
- 강점 3가지
- 약점 3가지
- 개선 제안 3가지 (우선순위 순)

대상: https://example-school-site.com
```

## 4. 코딩 프롬프트 작성법

### 학습 목표

- 코드 생성, 수정, 디버깅, 설명 요청을 위한 프롬프트를 작성할 수 있다
- 바이브코딩 도구에서 효과적으로 AI에게 코딩을 요청할 수 있다
- 코딩 프롬프트의 Before/After 차이를 이해한다

### 코딩 프롬프트의 4가지 유형

```

│                                          │
│  1. 생성 (Create)   새 코드 만들기       │
│  2. 수정 (Modify)   기존 코드 변경       │
│  3. 디버깅 (Debug)  오류 찾고 고치기     │
│  4. 설명 (Explain)  코드 이해하기        │
└──────────────────────────────────────────┘
```

## 5. 기획 문서 프롬프트

### 학습 목표

- AI를 활용하여 PRD(제품 요구사항 문서)를 작성할 수 있다
- 아이디어를 체계적인 기획서로 변환하는 프롬프트를 작성할 수 있다
- IA(정보 구조), 와이어프레임 등 기획 산출물을 AI로 생성할 수 있다

### 왜 기획 문서가 중요한가요?

바이브코딩 도구(Lovable, Cursor 등)에게 코드를 요청할 때, 기획 문서가 있으면 훨씬 좋은 결과를 얻을 수 있습니다:

```
아이디어만으로 요청:
  "학교 소개 사이트 만들어줘" → 어떤 페이지? 어떤 기능? 디자인은? → 기대와 다른 결과

기획 문서와 함께 요청:
  "이 PRD를 기반으로 사이트를 만들어줘" → 페이지 구조, 기능, 디자인 명확 → 원하는 결과
```

> **기획 문서는 AI에게 주는 "설계도"입니다.** 설계도 없이 집을 지으면 원하는 집이 나오지 않듯, 기획 문서 없이 앱을 만들면 원하는 결과를 얻기 어렵습니다.

### 단계 1: 아이디어를 PRD로 변환

#### 프롬프트 예시: 한 문장 아이디어에서 PRD 생성

```
당신은 IT 서비스 기획 전문가입니다.

아래 아이디어를 바탕으로 PRD(제품 요구사항 문서)를 작성해주세요.

아이디어: "우리 반 학생들이 독서 기록을 남기고 서로 추천할 수 있는 웹사이트"

PRD에 포함할 항목:
1. 프로젝트 개요 (한 문단)
2. 목표 사용자 (누가 쓸 것인가)
3. 핵심 기능 목록 (우선순위 표시: P0/P1/P2)
4. 페이지 구성 (각 페이지별 주요 요소)
5. 기술 요구사항 (프론트엔드, 저장 방식 등)
6. 디자인 가이드라인 (색상, 분위기, 참고 사이트)
7. MVP 범위 (첫 버전에서 반드시 포함할 기능)
8. 향후 확장 기능 (나중에 추가할 기능)

제약 조건:
- 바이브코딩 도구(Lovable)로 제작할 예정
- 데이터베이스: Supabase 또는 localStorage
- 배포: Lovable 기본 배포 사용
- 사용자 인증: 간단한 닉네임 방식 (소셜 로그인 제외)
```

### 단계 2: IA(정보 구조) 생성

#### 프롬프트 예시: PRD에서 IA 도출

```
아래 PRD를 바탕으로 정보 구조(IA)를 작성해주세요.

형식:
1. 사이트맵 (트리 구조, 텍스트 기반)
2. 각 페이지별:
   - 페이지명
   - URL 경로
   - 주요 섹션/컴포넌트
   - 사용자 행동(CTA)
3. 네비게이션 구조

[PRD 내용 붙여넣기]
```

#### 출력 예시:

```
사이트맵:
├── 홈 (/)
│   ├── 히어로 배너
│   ├── 최신 독서 기록
│   └── 추천 도서 TOP 5
├── 독서 기록 (/records)
│   ├── 내 독서 기록 목록
│   ├── 새 기록 작성 폼
│   └── 기록 상세 보기
├── 추천 도서 (/recommend)
│   ├── 추천 목록 (좋아요 순)
│   └── 추천하기 폼
└── 마이페이지 (/my)
    ├── 닉네임 설정
    ├── 내 기록 통계
    └── 읽은 책 수 뱃지
```

### 단계 3: 와이어프레임 설명 생성

```
아래 IA를 바탕으로 메인 페이지의 와이어프레임을 텍스트로 상세 설명해줘.

설명 형식:
- 상단부터 하단까지 순서대로
- 각 섹션의 위치, 크기, 포함 요소를 구체적으로
- 반응형 고려 (데스크탑/모바일 차이점)
- ASCII 아트로 대략적인 레이아웃 표현

이 설명을 바이브코딩 도구(Lovable)에 직접 넣어서 페이지를 생성할 예정입니다.

[IA 내용 붙여넣기]
```

### 기획 프롬프트 워크플로우

아이디어에서 완성까지의 프롬프트 체인:

```
1단계: 아이디어 → PRD
   "이 아이디어를 PRD로 만들어줘"

2단계: PRD → IA
   "이 PRD를 바탕으로 IA를 만들어줘"

3단계: IA → 와이어프레임 설명
   "이 IA를 바탕으로 메인 페이지 와이어프레임을 설명해줘"

4단계: 와이어프레임 → 코드 요청
   "이 와이어프레임 설명대로 HTML/CSS 코드를 만들어줘"

→ 각 단계의 결과를 다음 단계의 입력으로 사용!
```

> **실전 팁**: 기획 도우미 Gem을 활용하면 1~3단계를 더 빠르게 진행할 수 있습니다 → [Gemini Gem](https://gemini.google.com/gem/1yC701ZmuJgy_D4hkyNz0t50EqAnBj3kh)

### PRD 프롬프트 템플릿 (복사하여 사용)

```
당신은 IT 서비스 기획 전문가입니다.

아래 아이디어를 PRD(제품 요구사항 문서)로 작성해주세요.

아이디어: [한 문장으로 설명]

대상 사용자: [누구]
제작 도구: [Lovable / Cursor / Claude Code 등]
참고 사이트: [있으면 URL]

PRD에 반드시 포함할 항목:
1. 프로젝트 개요
2. 목표 사용자 페르소나
3. 핵심 기능 (우선순위: P0 필수 / P1 중요 / P2 희망)
4. 페이지별 상세 기능
5. 기술 스택 추천
6. 디자인 가이드 (색상, 폰트, 분위기)
7. MVP 범위
8. 제약 사항
```

## 6. 도구별 프롬프트 차이

### 학습 목표

- 각 바이브코딩 도구의 프롬프트 특성을 이해한다
- 도구에 맞는 프롬프트 스타일을 선택할 수 있다
- 같은 요청이라도 도구에 따라 프롬프트를 다르게 작성할 수 있다

### 도구별 프롬프트 스타일 비교표

| 항목 | ChatGPT / Claude | Lovable | Cursor | Claude Code | AntiGravity |
|------|------------------|---------|--------|-------------|-------------|
| **인터페이스** | 채팅 대화창 | 채팅 + 실시간 미리보기 | IDE 내 AI 패널 | CLI 터미널 | 에이전트 IDE |
| **프롬프트 성격** | 범용 질의응답 | 웹앱 제작 요청 | 코드 편집 지시 | 터미널 명령 + 코드 | 에이전트 위임 |
| **한국어 지원** | 매우 좋음 | 좋음 (UI는 영어) | 보통 | 좋음 | 보통 |
| **프롬프트 길이** | 자유로움 | 중간~긴 | 짧은~중간 | 중간~긴 | 중간 |
| **맥락 유지** | 대화 내 유지 | 프로젝트 전체 | 파일/프로젝트 | 프로젝트 전체 | 대화 내 유지 |
| **최적 프롬프트** | 범용 RCOTF | 기능 + UI 중심 | 코드 변경 중심 | 작업 지시 중심 | 목표 위임 중심 |

### 도구별 프롬프트 예시: 같은 요청을 다르게

아래는 "투두리스트 웹앱을 만들어줘"라는 동일한 요청을 각 도구에 맞게 작성한 예시입니다.

---

#### ChatGPT / Claude 에게 (범용 AI 챗봇)

```
React로 투두리스트 웹앱의 전체 코드를 작성해줘.

기능:
- 할 일 추가, 완료 토글, 삭제
- localStorage 저장
- 필터 (전체/진행중/완료)

조건:
- 하나의 코드 블록에 모든 코드 포함
- 주석으로 각 부분 설명
- 복사해서 바로 실행할 수 있도록
```

> 특징: 코드를 직접 받아서 내가 파일을 만들어야 함. 완성 코드를 요청.

---

#### Lovable 에게 (웹앱 빌더)

```
투두리스트 웹앱을 만들어줘.

주요 기능:
1. 할 일 추가 (입력창 + 추가 버튼)
2. 할 일 완료 체크 (체크박스, 완료 시 취소선)
3. 할 일 삭제 (각 항목에 삭제 버튼)
4. 필터: 전체 / 진행중 / 완료
5. 남은 할 일 개수 표시

디자인:
- 미니멀하고 깔끔한 스타일
- 메인 색상: #4A90D9 (파란색)
- 카드 형태의 중앙 레이아웃
- 모바일 반응형

데이터: localStorage에 저장하여 새로고침 시 유지
```

> 특징: UI/UX에 초점. 코드 언급 없이 원하는 기능과 디자인을 설명. Lovable이 알아서 기술을 선택.

---

#### Cursor 에게 (AI 코드 에디터)

```
이 프로젝트에 투두리스트 기능을 구현해줘.

파일 구조:
- src/components/TodoApp.tsx (메인 컴포넌트)
- src/components/TodoItem.tsx (개별 항목)
- src/components/TodoInput.tsx (입력 컴포넌트)
- src/hooks/useTodos.ts (상태 관리 커스텀 훅)

기능: CRUD + 필터링 + localStorage 연동
스타일: Tailwind CSS 사용
타입: TypeScript strict mode
```

> 특징: 파일 구조와 기술 스택을 구체적으로 지정. 개발자 관점에서 프로젝트 구조를 설계.

---

#### Claude Code 에게 (CLI 도구)

```
React + TypeScript + Tailwind CSS로 투두리스트 웹앱을 만들어줘.

요구사항:
- Vite로 프로젝트 초기화
- CRUD 기능 (추가/읽기/수정/삭제)
- localStorage 기반 데이터 저장
- 컴포넌트 분리 (TodoApp, TodoItem, TodoInput, TodoFilter)
- 커스텀 훅 (useTodos)으로 상태 관리

완료 후 dev 서버를 실행해줘.
```

> 특징: 프로젝트 초기화부터 실행까지 전체 워크플로우를 요청. 터미널 명령까지 포함.

---

#### AntiGravity 에게 (에이전트 IDE)

```
투두리스트 웹 애플리케이션을 처음부터 끝까지 만들어줘.

핵심 기능: 할 일 CRUD, 필터링, 로컬 저장
디자인: 깔끔하고 현대적인 UI
기술: 적절한 스택을 자유롭게 선택해

완성되면 브라우저에서 확인할 수 있게 해줘.
```

> 특징: 에이전트에게 자율적으로 위임. 기술 선택도 AI에게 맡기는 경우가 많음.

### 도구 선택 가이드

| 상황 | 추천 도구 | 이유 |
|------|----------|------|
| 코딩 경험 없음, 빠르게 웹사이트 만들고 싶음 | Lovable | 설치 없이 대화만으로 제작 |
| 코드를 배우며 개발하고 싶음 | Cursor | 코드를 직접 보고 제어 가능 |
| 터미널에 익숙하고 빠르게 작업하고 싶음 | Claude Code | CLI 기반 강력한 자동화 |
| 복잡한 프로젝트를 AI에게 위임하고 싶음 | AntiGravity | 에이전트가 자율적으로 작업 |
| 범용 질문/글쓰기/분석 | ChatGPT, Claude | 코딩 외 모든 작업에 활용 |

> **핵심**: 도구마다 AI가 기대하는 프롬프트 스타일이 다릅니다. Lovable에는 "무엇을 만들고 싶은지"를, Cursor에는 "어떤 코드를 어떻게 변경할지"를 중심으로 프롬프트를 작성하세요.

## 7. 반복 개선(Iteration)

### 학습 목표

- 한 번에 완벽한 결과를 기대하지 않고 반복 개선하는 방법을 익힌다
- 효과적인 후속 프롬프트(Follow-up) 기법을 배운다
- 대화 흐름을 설계하여 원하는 결과에 도달하는 전략을 세운다

### 왜 반복이 중요한가요?

프롬프트 엔지니어링의 가장 중요한 비밀은 다음과 같습니다:

```
"처음부터 완벽한 결과는 없다.
 좋은 결과는 반복에서 나온다."
```

전문가도 한 번에 원하는 결과를 얻지 못합니다. 차이는 **피드백을 주고 개선하는 속도**입니다.

#### 반복 개선의 사이클

```
┌────────────┐
│ 1. 프롬프트 │
│    작성     │
└──────┬─────┘
       ↓
┌────────────┐
│ 2. 결과    │
│    확인    │
└──────┬─────┘
       ↓
┌────────────┐     ┌────────────┐
│ 3. 평가    │ ──→ │ 만족하면   │ → 완성!
│            │     │ 종료       │
└──────┬─────┘     └────────────┘
       ↓ 불만족
┌────────────┐
│ 4. 피드백  │
│    작성    │
└──────┬─────┘
       ↓
   1번으로 돌아감
```

### 후속 프롬프트 유형 7가지

#### 유형 1: 방향 수정

```
좋아, 그런데 전체적인 톤이 너무 딱딱해.
좀 더 친근한 말투로 다시 써줘. 반말은 쓰지 말고,
"~해요" 체를 사용해줘.
```

#### 유형 2: 부분 수정

```
전체적으로 좋은데, 3번 항목을 수정해줘.
"환경 보호의 중요성" 부분을 삭제하고,
대신 "지역 사회 참여 활동" 내용으로 바꿔줘.
나머지는 그대로 유지해.
```

#### 유형 3: 확장 요청

```
이 내용에 다음을 추가해줘:
1. 각 항목에 실제 사례 1개씩 추가
2. 마지막에 "핵심 정리" 표 추가
3. 참고 자료 링크 3개 추가
```

#### 유형 4: 축소 요청

```
너무 길어. 현재 분량의 절반으로 줄여줘.
핵심 내용만 남기고, 부가 설명은 삭제해.
중요도가 낮은 항목(4번, 7번)은 통합하거나 제거해.
```

#### 유형 5: 형식 변경

```
이 내용을 표(테이블) 형식으로 변환해줘.
열: 항목 | 설명 | 예시 | 주의사항
```

#### 유형 6: 깊이 추가

```
2번 항목 "Few-shot 기법"을 더 자세히 설명해줘.
- 원리를 비유로 설명
- 실전 예시 3개 추가
- 잘 작동하지 않는 경우도 포함
```

#### 유형 7: 비교 요청

```
방금 결과를 A 버전이라고 하자.
같은 내용을 다른 스타일로 B 버전도 만들어줘:
- A 버전: 격식체, 교과서 스타일 (현재)
- B 버전: 구어체, 유튜브 대본 스타일

두 버전을 비교할 수 있게 나란히 보여줘.
```

### 효과적인 피드백 프롬프트 작성법

#### 나쁜 피드백 vs 좋은 피드백

| 나쁜 피드백 | 좋은 피드백 |
|------------|-----------|
| "마음에 안 들어" | "톤이 너무 격식적이야. '~합니다' 대신 '~해요' 체로 바꿔줘" |
| "다시 해줘" | "3번 항목의 예시를 교육 현장 사례로 바꿔줘" |
| "좀 더 좋게" | "도입부에 질문 형식의 hook을 추가해줘" |
| "이상해" | "두 번째 문단의 논리 흐름이 끊겨. A → B → C로 연결되게 수정해줘" |

#### 피드백 프롬프트 템플릿

```
방금 결과에서 다음을 수정해줘:

잘된 점 (유지할 것):
- [유지할 요소 1]
- [유지할 요소 2]

수정할 점:
1. [구체적 수정 사항 1]
2. [구체적 수정 사항 2]

추가할 점:
- [추가 요소]

나머지는 현재 그대로 유지해줘.
```

### 대화 흐름 설계

복잡한 프로젝트에서는 대화의 흐름을 미리 설계하면 효율적입니다:

```
프로젝트: 학급 홈페이지 제작

대화 1: 기획
  └→ "학급 홈페이지의 PRD를 만들어줘"
  └→ 결과 확인 → 피드백 → PRD 확정

대화 2: 구조
  └→ "확정된 PRD로 IA를 만들어줘"
  └→ 결과 확인 → 피드백 → IA 확정

대화 3: 디자인
  └→ "메인 페이지 디자인 가이드를 만들어줘"
  └→ 결과 확인 → 피드백 → 디자인 확정

대화 4: 구현
  └→ "확정된 기획/디자인으로 코드를 만들어줘"
  └→ 결과 확인 → 수정 요청 → 완성
```

### 반복 개선 실전 팁

| 팁 | 설명 |
|----|------|
| **좋은 것은 칭찬** | "이 부분은 좋아"라고 명시하면 AI가 해당 부분을 유지 |
| **한 번에 하나씩** | 여러 수정을 한 번에 요청하면 일부가 빠질 수 있음 |
| **번호로 지칭** | "3번 항목" "두 번째 표"처럼 명확히 지칭 |
| **비교 요청** | "A안과 B안을 만들어줘"로 선택지 확보 |
| **되돌리기** | "2번째 답변 버전으로 돌아가서 거기서 수정해줘" |
| **새 대화 시작** | 대화가 너무 길어지면 핵심만 정리하여 새 대화 시작 |

> **기억하세요**: 프롬프트 엔지니어링은 "한 방에 해결"이 아니라 "대화를 통한 개선"입니다. AI와 협력하여 점진적으로 결과를 다듬어가는 과정이 핵심입니다.

## 8. 흔한 실수 10가지

### 학습 목표

- 초보자가 자주 범하는 프롬프트 실수를 인식한다
- 각 실수의 문제점과 개선 방법을 이해한다
- 실수를 피하는 습관을 기른다

---

### 실수 1: 너무 막연하게 요청하기

**문제:**

```
좋은 글 써줘
```

**왜 문제인가?** "좋은"의 기준이 사람마다 다릅니다. AI는 무엇이 "좋은"인지 판단할 수 없습니다.

**개선:**

```
학부모 대상 뉴스레터에 실을 "겨울 독서 추천" 글을 써줘.
- 분량: 400자 내외
- 톤: 친근하고 실용적
- 포함: 학년별 추천 도서 3권씩
```

---

### 실수 2: 한 번에 너무 많이 요청하기

**문제:**

```
학교 웹사이트 전체를 만들어줘. 메인 페이지, 교사 소개, 학사일정,
공지사항, 갤러리, 온라인 상담 신청, 급식 메뉴, 학생 포트폴리오,
도서 대출 시스템, 성적 조회까지 모두 포함해.
```

**왜 문제인가?** 너무 많은 기능을 한 번에 요청하면 각각의 품질이 떨어지고, 전체 구조가 엉성해집니다.

**개선:**

```
학교 웹사이트를 단계별로 만들겠습니다.
Step 1: 먼저 메인 페이지와 교사 소개 페이지만 만들어줘.

메인 페이지 요구사항:
[구체적 요구사항]

교사 소개 페이지 요구사항:
[구체적 요구사항]
```

### 실수 3: 맥락 없이 요청하기

**문제:**

```
이메일 써줘
```

**왜 문제인가?** 누가, 누구에게, 왜, 어떤 톤으로 보내는 이메일인지 모름.

**개선:**

```
거래처 담당자에게 보내는 납기 지연 안내 이메일을 써줘.

맥락:
- 보내는 사람: 영업팀 과장
- 받는 사람: (주)한빛교육 김 대리
- 상황: 원자재 수급 문제로 납기가 5일 지연됨
- 톤: 정중하고 죄송한 마음을 전하되, 신뢰감 유지
- 포함: 사과, 지연 사유, 새로운 납기일, 보상(추가 할인) 제안
```

---

### 실수 4: 부정형으로만 요청하기

**문제:**

```
딱딱하게 쓰지 마. 재미없게 쓰지 마. 길게 쓰지 마.
```

**왜 문제인가?** "~하지 마"만 있으면 AI는 무엇을 해야 하는지 모릅니다. 하지 말라는 것만 알 뿐, 원하는 방향이 불분명합니다.

**개선:**

```
친근하고 유머러스한 톤으로 써줘.
중학생이 재미있게 읽을 수 있도록.
300자 이내로 간결하게.

(참고: 격식체나 학술적 표현은 피해줘)
```

> **팁**: "~하지 마"보다 "~해줘"를 먼저 쓰고, 필요하면 금지 사항을 추가하세요. 긍정형 지시가 항상 먼저!

---

### 실수 5: AI의 답변을 무조건 그대로 사용하기

**문제:**

AI가 생성한 결과를 검증 없이 그대로 복사하여 사용하는 것.

**왜 문제인가?**

- AI는 "그럴듯한 답변"을 생성하지만, 사실과 다를 수 있습니다 (할루시네이션)
- 특히 숫자, 날짜, 인물, 법률, 의학 정보는 반드시 확인 필요
- 코드도 문법적으로는 맞지만 논리적 오류가 있을 수 있음

**개선 방법:**

```
[결과 확인 체크리스트]
□ 사실 관계가 맞는가? (특히 숫자, 날짜, 고유명사)
□ 내 상황에 맞게 수정할 부분이 있는가?
□ 코드라면 실행해서 테스트했는가?
□ 민감한 정보(개인정보, 저작권 등)가 포함되어 있진 않은가?
```

### 실수 6: 결과 형식을 지정하지 않기

**문제:**

```
마케팅 전략 알려줘
```

**왜 문제인가?** AI가 긴 문단으로 답하는지, 목록으로 답하는지, 표로 답하는지 예측 불가.

**개선:**

```
마케팅 전략을 아래 형식으로 정리해줘:

| 전략 | 대상 | 채널 | 예산 | 기대 효과 |
|------|------|------|------|----------|
```

---

### 실수 7: 대화의 맥락을 무시하기

**문제:**

긴 대화를 나눈 후, 갑자기 전혀 다른 주제를 이전 맥락 위에서 요청하는 것.

**왜 문제인가?** AI는 이전 대화를 기반으로 답변하므로, 맥락이 섞이면 혼란스러운 결과가 나옵니다.

**개선:**

```
[새로운 주제를 시작할 때]

이전 대화와 별개의 새로운 요청입니다:
[새로운 요청]

또는: 새 대화(새 채팅)를 시작하세요.
```

---

### 실수 8: 전문 용어를 설명 없이 사용하기

**문제:**

```
CRUD 기능이 있는 SPA를 SSR로 구현하고 CI/CD 파이프라인 설정해줘
```

이 프롬프트 자체는 문제가 없지만, 본인이 이 용어들을 정확히 이해하지 못한 채 사용하면 결과를 평가할 수 없습니다.

**개선:**

```
다음 기능이 있는 웹 애플리케이션을 만들어줘:
- 데이터를 만들고(Create), 읽고(Read), 수정하고(Update), 삭제하는(Delete) 기능
- 페이지 새로고침 없이 화면이 전환됨
- 서버에서 미리 HTML을 만들어서 보내는 방식

각 기술 결정마다 왜 그렇게 했는지 초보자가 이해할 수 있게 설명 포함해줘.
```

### 실수 9: AI에게 결정을 전부 맡기기

**문제:**

```
적절하게 알아서 해줘
```

**왜 문제인가?** "적절하게"의 기준이 당신과 AI가 다를 수 있습니다.

**개선:**

```
색상은 파란색 계열(#2196F3 근처)로 해줘.
폰트 크기는 본문 16px, 제목 24px로.
레이아웃은 2단 구성 (좌측 메뉴, 우측 본문).
```

> **팁**: "알아서 해줘"를 쓸 수 있는 때는 정말 어떤 결과가 나와도 괜찮을 때뿐입니다. 대부분의 경우 구체적인 기준을 제시하는 것이 좋습니다.

### 실수 10: 한 번 실패하면 포기하기

**문제:**

```
(첫 번째 프롬프트에서 원하는 결과가 안 나옴)
→ "이 AI는 안 돼. 다른 AI를 써봐야지."
```

**왜 문제인가?** 프롬프트를 개선하면 같은 AI에서도 훨씬 좋은 결과를 얻을 수 있습니다.

**개선 전략:**

```
1. 결과에서 어떤 부분이 기대와 다른지 구체적으로 파악
2. 그 부분에 대한 추가 지시를 프롬프트에 반영
3. 2~3번 반복하면 대부분 원하는 결과에 도달

예시:
"방금 결과에서 톤이 너무 격식적이야.
'합니다' 체 대신 '해요' 체로 다시 작성해줘.
내용은 유지하되 더 친근하게."
```

### 실수 예방 체크리스트

프롬프트를 보내기 전에 아래를 확인하세요:

```
□ 1. 요청이 구체적인가? (누구, 무엇, 어떻게, 얼마나)
□ 2. 한 번에 요청하는 양이 적절한가? (3~5개 이내)
□ 3. 맥락(상황, 배경)을 충분히 설명했는가?
□ 4. "~해줘"가 "~하지 마"보다 많은가?
□ 5. 결과를 검증할 기준이 있는가?
□ 6. 출력 형식을 지정했는가?
□ 7. 대화 맥락이 맞는가? (새 주제면 새 대화?)
□ 8. 사용한 용어를 나도 이해하고 있는가?
□ 9. AI에게 핵심 결정을 맡기지 않았는가?
□ 10. 피드백으로 개선할 준비가 되어 있는가?
```

## 9. 실전 프로젝트

### 학습 목표

- 프롬프트만으로 웹 페이지를 제작하는 전 과정을 체험한다
- 이 교안에서 배운 모든 원칙을 실전에 적용한다
- 프롬프트 반복 개선의 실제 과정을 경험한다

### 프로젝트 소개: "나만의 자기소개 웹 페이지 만들기"

이 프로젝트에서는 AI와의 대화만으로 자기소개 웹 페이지를 만듭니다. 직접 코딩하지 않고, 프롬프트만 작성합니다.

```
최종 결과물: 자기소개 웹 페이지 (HTML 파일 1개)

포함 요소:
- 프로필 영역 (이름, 소개, 사진 자리)
- 자기소개 본문
- 관심사/취미 섹션
- 연락처/SNS 링크
- 반응형 디자인 (모바일 대응)
```

### Step 1: 기획 프롬프트 작성

먼저 AI에게 기획을 요청합니다:

```
당신은 웹 디자이너 겸 기획자입니다.

개인 자기소개 웹 페이지의 기획안을 만들어주세요.

대상: 교사 (교육 블로그에 연결할 자기소개 페이지)
분위기: 깔끔하고 전문적이면서도 따뜻한 느낌
색상: 네이비 블루(#1e3a5f) + 화이트 + 연한 그레이

기획안에 포함할 내용:
1. 페이지 구조 (위→아래 순서대로 섹션 설명)
2. 각 섹션에 들어갈 내용 항목
3. 디자인 스타일 (참고: Apple 스타일의 깔끔함)
4. 모바일에서의 레이아웃 차이
```

> 이 단계에서 기획안을 확인하고, 필요하면 수정합니다.

### Step 2: 코드 생성 프롬프트 작성

기획이 확정되면 코드를 요청합니다:

```
위 기획안을 바탕으로 자기소개 웹 페이지를 HTML/CSS/JavaScript로 만들어줘.

요구사항:
1. 파일: index.html 하나에 CSS와 JS 모두 포함
2. 반응형: 데스크탑(1200px) / 태블릿(768px) / 모바일(480px)

섹션별 상세:

[히어로 섹션]
- 배경: 그라디언트 (네이비 → 약간 밝은 블루)
- 프로필 이미지: 원형, 150px, 테두리 흰색
  (이미지 없으면 이니셜 표시)
- 이름: "김바이브" (큰 글씨, 흰색)
- 한줄 소개: "아이들과 함께 성장하는 교사" (작은 글씨, 연한 파랑)

[소개 섹션]
- 제목: "안녕하세요!"
- 본문: 3~4문장의 자기소개 (placeholder 텍스트)
- 아이콘 포인트 3개: 경력 10년 / 과학 전공 / AI 교육 관심

[관심사 섹션]
- 카드 형식으로 4개
- 각 카드: 아이콘 + 제목 + 설명 1줄
- hover 시 약간 위로 떠오르는 효과

[연락처 섹션]
- 이메일, 블로그, GitHub 아이콘 링크
- 배경: 연한 그레이

[푸터]
- 저작권 표시
- "Powered by 바이브코딩" 텍스트

디자인 조건:
- 폰트: 시스템 폰트 (Pretendard fallback)
- 스크롤 시 네비게이션 바 고정
- 부드러운 스크롤 (smooth scroll)
- 각 섹션 진입 시 fade-in 애니메이션
```

### Step 3: 결과 확인 및 피드백

코드를 받으면 브라우저에서 열어보고, 피드백합니다:

```
코드 확인 결과 다음을 수정해줘:

유지할 점:
- 전체 레이아웃 구조는 좋음
- 색상 조합 맘에 듦
- 카드 hover 효과 좋음

수정할 점:
1. 히어로 섹션 높이가 너무 높음 → 80vh에서 60vh로 줄여줘
2. 모바일에서 카드가 2열로 나오는데 → 1열로 변경
3. 폰트 크기가 전체적으로 너무 작음 → 본문 18px로 키워줘
4. 네비게이션 바에 현재 섹션 하이라이트 기능 추가

추가 요청:
- 상단에 "Back to top" 버튼 추가 (하단 스크롤 시 나타남)
- 다크모드 토글 버튼 추가
```

### Step 4: 최종 점검 및 완성

```
거의 완성이야! 마지막으로 아래 사항만 확인하고 최종 코드를 줘:

체크리스트:
1. HTML 유효성: 모든 태그가 올바르게 닫혔는지
2. 접근성: alt 텍스트, aria-label 등 접근성 속성 포함
3. SEO: meta description, og 태그 포함
4. 성능: 불필요한 코드 제거, CSS 정리
5. 개인정보 placeholder 텍스트가 실제 정보가 아닌지 확인

위 항목을 모두 반영한 최종 완성 코드를 줘.
```

### Step 5: 배포 (선택사항)

완성된 코드를 배포하는 방법:

```
방법 1: GitHub Pages
- GitHub에 저장소를 만들고 index.html 업로드
- Settings → Pages에서 배포

방법 2: Lovable에 업로드
- Lovable에서 새 프로젝트 생성
- "이 HTML 코드를 기반으로 프로젝트를 시작해줘" 요청

방법 3: Netlify Drop
- netlify.com/drop 에 파일을 드래그 앤 드롭
- 즉시 배포 완료
```

### 프로젝트 회고 프롬프트

프로젝트 완료 후 자신의 프롬프트를 되돌아봅니다:

```
이 대화에서 내가 사용한 프롬프트들을 분석해줘:

1. 어떤 원칙(구체성, 역할, 예시, 단계, 제약)을 잘 활용했는지
2. 어떤 원칙이 부족했는지
3. 가장 효과적이었던 프롬프트는 어느 것이었는지
4. 다음에 비슷한 프로젝트를 할 때 개선할 점

표 형식으로 정리해줘.
```

## 10. FAQ & 다음 단계

### 자주 묻는 질문

#### Q1: 프롬프트를 한국어로 써도 되나요?

**A:** 네, 대부분의 AI 모델(ChatGPT, Claude, Gemini 등)은 한국어를 잘 이해합니다. 다만 몇 가지 팁이 있습니다:

| 상황 | 추천 언어 |
|------|----------|
| 일반적인 질문/글쓰기 | 한국어 |
| 코드 생성 | 한국어 + 기술 용어는 영어 |
| Lovable/Cursor 사용 | 한국어 (UI는 영어지만 프롬프트는 한국어 OK) |
| 영어 콘텐츠 생성 | 영어 프롬프트 추천 |

> 코드 변수명이나 기술 스택 이름은 영어로 적는 것이 좋습니다. 예: "React 컴포넌트를 만들어줘" (O) / "리액트 구성요소를 만들어줘" (X)

---

#### Q2: 어떤 AI 모델이 프롬프트 엔지니어링에 가장 좋나요?

**A:** 모델마다 강점이 다릅니다:

| 모델 | 강점 | 추천 상황 |
|------|------|----------|
| **Claude** | 긴 문맥 이해, 정확한 지시 수행, 안전성 | 문서 작성, 코딩, 분석 |
| **ChatGPT** | 범용성, 플러그인 생태계, 이미지 생성 | 범용 작업, 브레인스토밍 |
| **Gemini** | Google 서비스 연동, 멀티모달 | 검색 연동, 스프레드시트 |

> 어떤 모델을 쓰든 이 교안의 5가지 원칙은 동일하게 적용됩니다.

---

#### Q3: 프롬프트가 길면 더 좋은 결과가 나오나요?

**A:** 반드시 그렇지는 않습니다. 핵심은 "길이"가 아니라 "정보의 질"입니다.

```
나쁜 예 (길지만 비효과적):
"음... 그러니까... 내가 원하는 건... 블로그 같은 거?
글을 쓰는 건데... 아무래도 사람들이 좋아할 만한...
그런 느낌의 글이면 좋겠어... 잘 부탁해..."

좋은 예 (짧지만 효과적):
"교사 대상 교육 블로그에 올릴 'AI 수업 활용법' 글.
800자, 실전 팁 중심, '해요' 체."
```

핵심 정보(대상, 주제, 형식, 분량, 톤)만 명확하면 짧아도 좋은 결과를 얻을 수 있습니다.

---

#### Q4: 같은 프롬프트인데 매번 다른 결과가 나와요. 왜 그런가요?

**A:** AI 모델은 기본적으로 확률 기반으로 텍스트를 생성합니다. 같은 프롬프트라도 매번 약간 다른 결과가 나올 수 있습니다.

일관된 결과를 원한다면:
- 더 구체적인 프롬프트 작성 (변동 여지 줄이기)
- 출력 형식을 엄격하게 지정
- 예시(Few-shot)를 제공하여 패턴 고정
- 일부 AI 도구의 "temperature" 설정을 낮추기 (가능한 경우)

---

#### Q5: 프롬프트에 개인정보를 넣어도 안전한가요?

**A:** 주의가 필요합니다.

| 항목 | 권장 |
|------|------|
| 실명, 전화번호, 주소 | 가능하면 가명이나 placeholder 사용 |
| 회사 기밀 정보 | 넣지 않는 것을 강력 권장 |
| 학생 정보 | 절대 넣지 마세요 (개인정보보호법) |
| 일반적인 아이디어, 질문 | 대부분 안전 |

> 민감한 정보는 "[이름]", "[학교명]" 같은 placeholder로 대체하세요.

---

#### Q6: 바이브코딩 도구에서 프롬프트를 잘 쓰면 코딩을 안 배워도 되나요?

**A:** 프롬프트만으로도 많은 것을 만들 수 있지만, 기본적인 이해가 있으면 더 좋습니다.

```
프롬프트만으로 가능:  간단한 웹사이트, 블로그, 포트폴리오, 이벤트 페이지
기초 이해 + 프롬프트: 더 복잡한 기능, 버그 수정, 커스터마이징
코딩 지식 + 프롬프트: 전문적인 웹 애플리케이션, 데이터베이스 연동
```

> 코딩을 "공부"하기보다, 프롬프트를 쓰면서 AI가 만든 코드를 하나씩 이해해가는 방식을 추천합니다. 자연스럽게 배우는 것이 가장 효과적입니다.

---

#### Q7: 프롬프트를 저장하고 관리하는 좋은 방법이 있나요?

**A:** 좋은 프롬프트는 반드시 저장하세요. 추천 관리 방법:

| 방법 | 장점 | 추천 대상 |
|------|------|----------|
| **메모장/노션** | 간단, 검색 편리 | 개인 사용 |
| **스프레드시트** | 분류, 태그, 평가 가능 | 체계적 관리 |
| **GitHub Gist** | 버전 관리, 공유 가능 | 개발자 |
| **프롬프트 관리 도구** | 전문 기능 (PromptBase 등) | 고급 사용자 |

프롬프트 저장 시 포함할 정보:
```
- 프롬프트 내용
- 사용한 AI 모델
- 날짜
- 결과 만족도 (1~5)
- 사용 상황/목적
- 태그 (글쓰기, 코딩, 기획 등)
```

#### Q8: 프롬프트 엔지니어링을 더 깊이 배우려면 어디서 공부하나요?

**A:** 추천 학습 경로:

```
1단계 (이 교안): 기본 원칙 이해, 실습
   ↓
2단계 (중급자편): 고급 기법 (시스템 프롬프트, 멀티턴 전략, 평가 방법)
   ↓
3단계 (개발자편): API 활용, 프롬프트 자동화, 에이전트 설계
```

추천 외부 리소스:
- Anthropic Prompt Engineering Guide: Claude 공식 가이드
- Prompting Guide (promptingguide.ai): 기법별 체계적 정리
- Learn Prompting (learnprompting.org): 무료 온라인 강좌
- OpenAI Cookbook: 실전 예제 모음
