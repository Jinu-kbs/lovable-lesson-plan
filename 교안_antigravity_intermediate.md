## 2. 에이전트 워크플로우

### Manager View vs Editor View

안티그래비티에는 두 가지 주요 화면이 있습니다:

| 화면 | 역할 | 사용 시점 |
|------|------|----------|
| **Manager View** | 에이전트에게 작업을 지시하고 모니터링 | 새 작업 요청, 수정 지시, 진행 상황 확인 |
| **Editor View** | VS Code 기반 코드 편집 환경 | 코드 직접 확인, 세부 수정, 파일 탐색 |

### Plan Mode vs Fast Mode

| 모드 | 특징 | 추천 상황 |
|------|------|----------|
| **Plan Mode** | 실행 전 계획서(Plan Artifact) 생성 → 검토 → 승인 후 실행 | 새 프로젝트, 큰 변경, 중요한 작업 |
| **Fast Mode** | 즉시 실행, 계획 단계 생략 | 작은 수정, 간단한 추가, 빠른 변경 |

### 에이전트 워크플로우 5단계

```
1단계: 작업 정의
  ↓ PRD 또는 상세 프롬프트 작성
2단계: 에이전트에게 전달
  ↓ Manager View → Plan Mode 선택 → Send
3단계: 계획 검토
  ↓ Plan Artifact 확인 → 승인 또는 수정 요청
4단계: 실행 모니터링
  ↓ Artifacts 탭에서 진행 상황 확인
5단계: 결과 검증
  ↓ Editor View에서 코드 확인 + 브라우저 테스트
```

> **활용 가이드**: "중급자는 Plan Mode를 기본으로 사용하되, 작은 수정에는 Fast Mode를 활용하세요. 모드 전환에 익숙해지는 것이 효율의 핵심입니다." — 활용 가이드

## 4. 기획 문서 작성 실습

### PRD / IA / ERD 개념

| 문서 | 역할 | 내용 |
|------|------|------|
| **PRD** | 무엇을 만들 것인가? | 프로젝트 목적, 기능 목록, 디자인 요구사항 |
| **IA** | 어떻게 구조를 잡을 것인가? | 페이지 구성, 네비게이션 흐름, 컴포넌트 트리 |
| **ERD** | 데이터를 어떻게 관리할 것인가? | 테이블 구조, 관계, 필드 정의 |

### Plan Mode와 기획 문서 연동

기획 문서를 에이전트에게 전달하면 Plan Mode에서 더 정확한 계획을 세울 수 있습니다:

1. PRD를 먼저 작성 (AI 도우미 활용)
2. Manager View에서 PRD를 붙여넣기
3. Plan Mode로 전송
4. 에이전트가 PRD 기반 Plan 생성
5. Plan 검토 후 승인

### PRD 작성 프롬프트

AI에게 PRD 작성을 요청하세요:

```
교사용 학습 관리 대시보드의 PRD를 작성해줘.

포함할 내용:
1. 프로젝트 개요 (목적, 대상 사용자)
2. 핵심 기능 목록 (우선순위별)
3. 페이지별 기능 상세
4. 디자인 요구사항 (색상, 레이아웃)
5. 기술 제약사항

대상: 초등학교 교사
주요 기능: 학생 관리, 출석 체크, 성적 입력, 주간 계획
```

### IA 작성 프롬프트

```
위 PRD를 기반으로 IA(정보 구조)를 작성해줘.

포함할 내용:
1. 사이트맵 (전체 페이지 구조)
2. 네비게이션 흐름
3. 각 페이지의 주요 컴포넌트
4. 모바일/데스크톱 레이아웃 차이
```

### ERD 작성 프롬프트

```
위 PRD를 기반으로 ERD(데이터 구조)를 작성해줘.

포함할 내용:
1. 필요한 테이블 목록
2. 각 테이블의 필드 (이름, 타입, 설명)
3. 테이블 간 관계 (1:N, N:M)
4. 인덱스 권장사항
```

## 5. 안티그래비티 심화 활용

### Rules 커스터마이징

Rules는 에이전트가 따라야 할 규칙을 정의합니다. 두 가지 수준이 있습니다:

| 수준 | 적용 범위 | 설정 위치 |
|------|----------|----------|
| **Global Rules** | 모든 워크스페이스에 적용 | 앱 설정 |
| **Workspace Rules** | 특정 프로젝트에만 적용 | 워크스페이스 설정 / `.agents/` 디렉토리 |

### Workflows 활용

Workflows는 반복적인 작업을 자동화하는 레시피입니다:

```
# Workflow 예시: 새 기능 추가

1. 기능 요구사항을 Plan Mode로 분석
2. 필요한 컴포넌트 파일 생성
3. 기존 코드와의 통합 포인트 확인
4. 테스트 코드 작성
5. 브라우저에서 동작 확인
```

### 모델 선택 전략

안티그래비티는 여러 AI 모델을 지원합니다:

| 모델 | 특징 | 추천 용도 |
|------|------|----------|
| **Gemini 3 Pro** | 빠르고 정확, 기본 모델 | 일반적인 개발 작업 |
| **Claude Sonnet** | 긴 컨텍스트, 복잡한 작업 | 대규모 리팩토링, 문서 분석 |
| **GPT 모델** | 다양한 지식 | 특수 도메인 작업 |

### 컨텍스트 윈도우 관리

에이전트가 한 번에 처리할 수 있는 정보량(컨텍스트 윈도우)에는 한계가 있습니다:

- **작은 작업으로 분할**: 한 번에 너무 많은 것을 요청하지 마세요
- **중간 결과 확인**: 큰 작업은 단계별로 진행
- **명확한 파일 지정**: "src/components/Header.tsx를 수정해줘"처럼 구체적으로

## 6. 실전 프로젝트: 멀티 에이전트 활용 웹앱

### 프로젝트 예시: 팀 대시보드

팀원들의 업무 현황을 한눈에 볼 수 있는 대시보드를 만들어봅시다.

### Step 1: 기획 문서 전달

Manager View에서 PRD를 전달합니다:

```
팀 업무 관리 대시보드를 만들어줘.

## 핵심 기능
1. 팀원 목록 및 프로필
2. 업무 보드 (칸반 스타일: 할 일 / 진행 중 / 완료)
3. 일정 캘린더
4. 간단한 통계 (완료율, 팀원별 업무 수)

## 기술 스택
- React + TypeScript
- Tailwind CSS
- localStorage로 데이터 저장 (DB 없이)

## 디자인
- 깔끔한 모던 스타일
- 다크/라이트 모드 전환
- 반응형 (모바일 대응)
```

### Step 2: Plan 검토 및 수정

에이전트가 생성한 Plan을 검토합니다:
- 파일 구조가 적절한지 확인
- 기능 우선순위가 맞는지 확인
- 누락된 기능이 없는지 확인

필요시 피드백:
```
Plan을 확인했어.
칸반 보드에 드래그앤드롭 기능을 추가해줘.
그리고 통계 부분에 차트를 넣어줘.
```

### Step 3: 결과 확인 및 반복 개선

1. 브라우저에서 결과물 확인
2. 기능 테스트 (CRUD, 필터링 등)
3. 추가 수정 사항을 Fast Mode로 빠르게 적용
4. 디자인 미세 조정

## 8. 데이터베이스 연결

### Supabase 연동 (MCP 활용)

안티그래비티는 MCP(Model Context Protocol)를 통해 Supabase와 연동할 수 있습니다:

```
이 프로젝트에 Supabase를 연동해줘.
MCP를 사용해서 Supabase에 접속하고,
아래 테이블을 생성해줘:

- users: id, name, email, role
- tasks: id, title, description, status, assignee_id
- comments: id, task_id, user_id, content, created_at
```

### Firebase 연동

```
Firebase Realtime Database 또는 Firestore를 연동해줘.
로그인 기능도 Firebase Auth로 추가해줘.
```

### 에이전트 기반 DB 셋업

에이전트에게 데이터베이스 설정을 통째로 맡길 수 있습니다:

```
이 프로젝트에 사용자 인증과 데이터 저장 기능을 추가해줘.
Supabase를 사용하고, 아래 기능을 구현해줘:
1. 이메일/비밀번호 로그인
2. 사용자 프로필 관리
3. CRUD 기능 (생성/조회/수정/삭제)
```

## 9. 외부 서비스 연동

### MCP 서버 연결

MCP(Model Context Protocol)는 에이전트가 외부 서비스와 통신하는 표준 프로토콜입니다:

| 서비스 | MCP 서버 | 용도 |
|--------|---------|------|
| Supabase | supabase-mcp | 데이터베이스, 인증 |
| GitHub | github-mcp | 저장소 관리 |
| Figma | figma-mcp | 디자인 가져오기 |
| Slack | slack-mcp | 알림 연동 |

### Stripe 결제 연동

## 10. 문제가 생겼을 때 (트러블슈팅)

### 상황 1: 에이전트가 무한 루프에 빠져요

1. 작업을 중지 (Stop 버튼)
2. 문제 상황을 구체적으로 설명하여 새로 요청
3. 작업 범위를 줄여서 단계별로 진행

### 상황 2: 컨텍스트 오버플로우

에이전트가 "너무 많은 정보"라고 할 때:
1. 작업을 작은 단위로 분할
2. 관련 파일만 명시적으로 지정
3. 새 대화 세션에서 다시 시작

### 상황 3: 병합 충돌 (코드 충돌)

에이전트와 수동 수정이 충돌할 때:
1. Editor View에서 충돌 부분 확인
2. 에이전트에게 충돌 해결 요청
3. Git 히스토리에서 이전 버전 복원 가능

### 상황 4: 브라우저 에이전트 오류

1. Chrome 확장 업데이트 확인
2. Chrome 재시작
3. 안티그래비티 앱 재시작

### 상황 5: 속도 제한에 걸렸어요

1. 5시간마다 자동 리셋 → 잠시 기다리기
2. 기다리는 동안 기획 문서 보완
3. 다른 모델로 전환 시도

### 상황 6: 에이전트가 잘못된 방향으로 진행해요

1. 즉시 중지
2. Plan Mode로 전환하여 계획부터 재확인
3. 더 구체적인 프롬프트로 방향 수정
4. Rules에 제약사항 추가

## 11. 자주 묻는 질문 (FAQ)

### Q1. 멀티 에이전트를 사용하면 코드가 꼬이지 않나요?

안티그래비티는 에이전트 간 작업을 자동으로 조율합니다. 하지만 같은 파일을 동시에 수정하면 충돌이 발생할 수 있으므로, 작업 영역을 분리하는 것이 좋습니다.

### Q2. 에이전트가 만든 코드의 소유권은?

사용자에게 있습니다. 안티그래비티로 생성한 모든 코드는 사용자가 소유합니다.

### Q3. 오프라인에서도 작업할 수 있나요?

AI 에이전트 기능은 인터넷이 필요합니다. 하지만 Editor View에서 코드를 직접 수정하는 것은 오프라인에서도 가능합니다.

### Q4. 데이터 프라이버시는 안전한가요?

Google의 데이터 보호 정책이 적용됩니다. 프리뷰 기간에는 프로젝트 코드가 Google 서버를 경유합니다. 민감한 데이터(API 키, 비밀번호 등)는 환경변수로 관리하세요.

### Q5. 러버블에서 만든 프로젝트를 안티그래비티로 가져올 수 있나요?

네. 러버블에서 프로젝트를 GitHub에 내보낸 후, 안티그래비티에서 해당 저장소를 워크스페이스로 열면 됩니다.

### Q6. 한국어 프롬프트가 잘 작동하나요?

네. 에이전트에게 한국어로 지시할 수 있으며, 한국어 프롬프트도 잘 이해합니다. 다만 UI(메뉴, 버튼)는 영어입니다.

### Q7. 여러 프로젝트를 동시에 관리할 수 있나요?

네. 여러 워크스페이스를 만들어 프로젝트별로 독립적으로 관리할 수 있습니다.

### Q8. Pro 계정이 필요한 기능이 있나요?

현재 퍼블릭 프리뷰 기간에는 모든 기능이 무료입니다. 향후 유료 전환 시 일부 기능이 Pro 전용으로 변경될 수 있습니다.

## 12. 에이전트 최적화 전략

### 효율적 프롬프팅

| 방식 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| **구체성** | "이쁘게 만들어줘" | "배경을 #F5F5F5로, 버튼을 둥글게 (border-radius: 8px)" |
| **범위** | "전체 앱을 다시 만들어줘" | "Header 컴포넌트의 네비게이션만 수정해줘" |
| **단계** | "로그인부터 결제까지 전부" | "1단계: 로그인 UI 먼저 만들어줘" |
| **파일 지정** | "코드 수정해줘" | "src/components/Dashboard.tsx에서 차트 부분을 수정해줘" |

### Plan Mode 활용 전략

1. **새 기능 추가**: Plan Mode → 계획 검토 → 승인 → 실행
2. **기존 기능 수정**: Fast Mode로 빠르게 적용
3. **대규모 리팩토링**: Plan Mode → 단계별 실행 → 중간 검토
4. **버그 수정**: Fast Mode로 즉시 수정

### Rules 자동화

프로젝트에 맞는 Rules를 미리 설정하면 매번 지시할 필요 없이 에이전트가 규칙을 따릅니다:

```
# 프로젝트 Rules

## 코드 스타일
- TypeScript strict 모드 사용
- ESLint + Prettier 적용
- 컴포넌트는 함수형으로 작성

## 커밋 규칙
- 기능 추가: feat: 설명
- 버그 수정: fix: 설명
- 리팩토링: refactor: 설명

## 테스트
- 핵심 비즈니스 로직에는 반드시 테스트 작성
- 테스트 프레임워크: Vitest 또는 Jest
```
